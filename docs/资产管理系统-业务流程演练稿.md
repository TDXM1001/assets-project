## Problem Statement

前面的文档已经把系统边界、表结构、权限、接口和开发计划都定得差不多了。现在还差最后一块特别关键的验证：

不是“设计看起来对不对”，而是“真实业务流能不能在这套模型里顺畅表达出来”。

这一版不继续加新表，也不继续加新接口，而是拿 3 条最有代表性的流程做完整演练：

1. 入库并直接领用
2. 跨部门调拨
3. 盘点发现位置不符

每条流程都从下面几个角度走一遍：

1. 谁发起
2. 业务单据怎么建
3. 单据状态怎么变化
4. 资产主档怎么变化
5. 事件流水怎么落
6. 哪些点最容易出错

## Constraints

1. 演练基于前面的表结构和权限模型，不引入新表。
2. 仍以第一阶段 MVP 为边界，不考虑采购系统、BPM、扫码枪、财务折旧。
3. 重点验证模型是否合理，不追求覆盖所有异常分支。

## Premises

1. 资产主档：`asset_info`
2. 通用单据：`asset_operate_order` + `asset_operate_order_item`
3. 盘点任务：`asset_inventory_task` + `asset_inventory_task_item`
4. 历史流水：`asset_event_log`
5. 单据状态与资产状态分离

## Flow 1: 入库并直接领用

### 业务背景

研发部新采购了 3 台笔记本电脑，到货当天不进入闲置库存，而是直接分配给 3 名研发工程师。

### 参与角色

1. 资产管理员
2. 部门主管
3. 资产使用人

### 业务目标

让 3 台设备从“新到货”直接变成“在用”，并且在系统中明确：

1. 资产编码
2. 分类
3. 所属部门
4. 当前责任人
5. 存放位置
6. 完整事件流水

### 推荐处理方式

这条流程建议拆成两个动作，而不是一张巨型单据全做完：

1. 入库建账
2. 领用发放

这样更符合实际，也更容易审计。

### Step 1: 创建入库单

单据头：

1. `order_type = INBOUND`
2. `order_status = DRAFT`
3. `apply_user_id = 资产管理员`
4. `apply_dept_id = 资产管理部门`

单据明细：

1. 3 条 `asset_operate_order_item`
2. 每条明细对应 1 台设备
3. `before_status` 为空
4. `after_status = IDLE`

### Step 2: 完成入库单

当入库单完成时：

1. 在 `asset_info` 新增 3 条资产主档
2. 状态为 `IDLE`
3. `use_org_dept_id = 研发部`
4. `current_user_id = null`
5. `current_location_id = 研发部仓位/收货区`

同时写入 3 条事件流水：

1. `event_type = INBOUND`
2. `source_order_type = INBOUND`
3. `before_snapshot = null`
4. `after_snapshot = 新资产快照`

### Step 3: 创建领用单

单据头：

1. `order_type = ASSIGN`
2. `order_status = DRAFT`
3. `apply_user_id = 资产管理员` 或由使用人发起
4. `to_dept_id = 研发部`

单据明细：

1. 每条明细对应 1 台笔记本
2. `before_status = IDLE`
3. `after_status = IN_USE`
4. `before_user_id = null`
5. `after_user_id = 对应员工`
6. `before_location_id = 收货区`
7. `after_location_id = 员工工位`

### Step 4: 审批与完成

如果流程要求部门主管确认：

1. 单据状态：`DRAFT -> SUBMITTED -> APPROVED -> DONE`

完成后回写 `asset_info`：

1. `asset_status = IN_USE`
2. `current_user_id = 对应员工`
3. `current_location_id = 员工工位`
4. `use_org_dept_id = 研发部`

同时写事件流水：

1. `event_type = ASSIGN`
2. 记录变更前后责任人和位置

### 演练结论

这条流程验证了三件事：

1. 入库和领用必须拆开，主档和单据模型才清晰
2. `asset_info` 适合保存当前快照
3. `asset_event_log` 适合保存完整历史

### 这条流程最容易出错的地方

1. 想省事，把“入库并领用”做成直接新增 `asset_info`
2. 结果没有入库事件，后面追溯采购到货会断

正确做法：

1. 仍然保留入库动作
2. 领用作为第二步动作

## Flow 2: 跨部门调拨

### 业务背景

市场部一台投影仪调拨给培训中心使用，责任人和位置都要变，但资产本身并没有离开“在用”状态。

### 参与角色

1. 资产管理员
2. 来源部门主管
3. 目标部门主管

### 业务目标

完成以下变化：

1. 所属部门从市场部变成培训中心
2. 当前责任人从市场专员变成培训专员
3. 存放位置从市场会议室变成培训教室
4. 资产状态仍然保持 `IN_USE`

### Step 1: 创建调拨单

单据头：

1. `order_type = TRANSFER`
2. `from_dept_id = 市场部`
3. `to_dept_id = 培训中心`
4. `from_user_id = 市场专员`
5. `to_user_id = 培训专员`
6. `from_location_id = 市场会议室`
7. `to_location_id = 培训教室`

单据状态：

1. `DRAFT`

单据明细：

1. 对应 1 台投影仪
2. `before_status = IN_USE`
3. `after_status = IN_USE`
4. 前后部门、责任人、位置全部记录

### Step 2: 调拨审批

这类流程建议至少双确认：

1. 来源部门确认可移交
2. 目标部门确认接收

一期如果不做复杂审批流，可以简化成：

1. `SUBMITTED`
2. `APPROVED`

审批细节由业务字段承载，不额外引入复杂节点表。

### Step 3: 完成调拨

完成后回写 `asset_info`：

1. `use_org_dept_id = 培训中心`
2. `current_user_id = 培训专员`
3. `current_location_id = 培训教室`
4. `asset_status` 保持 `IN_USE`

写事件流水：

1. `event_type = TRANSFER`
2. `before_snapshot` 包含原部门、原责任人、原位置
3. `after_snapshot` 包含新部门、新责任人、新位置

### 演练结论

这条流程验证了一个很重要的点：

资产变化不一定是状态变化，也可能只是归属变化。

这说明：

1. `asset_operate_order_item` 必须保留 `before_* / after_*`
2. 不能只靠 `asset_status` 表达所有业务变化

### 这条流程最容易出错的地方

1. 只更新部门，不更新责任人和位置
2. 或者只更新主档，不写事件流水

后果：

1. 台账看起来变了
2. 但审计时完全不知道为什么变、谁改的

## Flow 3: 盘点发现位置不符

### 业务背景

财务部盘点时发现一台账面登记在“财务资料室”的打印机，实际被放在“财务开放办公区”。

资产没有丢失，也没有换责任人，只是位置不一致。

### 参与角色

1. 盘点执行人
2. 部门主管
3. 资产管理员

### 业务目标

完成以下事情：

1. 记录这次差异
2. 明确差异类型是“位置不符”
3. 在处理完成后更新资产位置
4. 保留盘点记录和事件流水

### Step 1: 发起盘点任务

任务头：

1. `task_scope_type = DEPT`
2. `target_dept_id = 财务部`
3. `task_status = DRAFT`

启动后：

1. `task_status = RUNNING`
2. 系统根据财务部名下资产生成盘点明细

### Step 2: 执行盘点

盘点明细针对该打印机写入：

1. `expected_location_id = 财务资料室`
2. `actual_location_id = 财务开放办公区`
3. `expected_user_id = 财务助理`
4. `actual_user_id = 财务助理`
5. `inventory_result = LOCATION_DIFF`
6. `process_status = 待处理`

### Step 3: 差异处理

资产管理员确认：

1. 资产确实还在财务部
2. 责任人没变
3. 只是位置记录过期

处理动作：

1. 将 `asset_info.current_location_id` 更新为实际位置
2. 将盘点明细 `process_status` 改为已处理
3. 记录处理说明

### Step 4: 写事件流水

这里建议仍写 1 条事件流水：

1. `event_type = INVENTORY`
2. `event_desc = 盘点发现位置不符并已更正`
3. `before_snapshot` 记录账面位置
4. `after_snapshot` 记录纠正后位置

### Step 5: 完成盘点任务

当所有差异都处理完：

1. `task_status = FINISHED`
2. 汇总 `summary_ok`
3. 汇总 `summary_diff`

### 演练结论

这条流程验证了：

1. 盘点不是普通单据，它更像“批量校验任务”
2. 盘点差异不是全都要走调拨或领用流程
3. `asset_inventory_task_item` 必须记录账面值和现场值

### 这条流程最容易出错的地方

1. 盘点时直接改主档，不保留差异记录
2. 结果后面只能看到“位置已经对了”，却看不到“曾经盘点有差异”

正确做法：

1. 先落盘点差异
2. 再落处理动作
3. 最后再回写主档和事件流水

## Cross-Flow Findings

把这 3 条流程放在一起看，可以验证出当前模型最关键的几个正确点。

### 1. `asset_info` 只存当前值是对的

因为三条流程里：

1. 入库并领用
2. 调拨
3. 盘点纠偏

最终都只需要一个“当前真实状态”。

### 2. `asset_event_log` 必须存在

因为三条流程里都需要回答：

1. 以前是什么
2. 为什么变
3. 谁操作的

没有流水，这些问题都答不出来。

### 3. 通用业务单据模型是够用的

因为：

1. 入库
2. 领用
3. 调拨

都可以用 `asset_operate_order + item` 表达。

### 4. 盘点必须独立

因为盘点的核心不是“发起业务动作”，而是“发现差异并处理差异”。

这和入库、领用、调拨天然不同。

## Risks Confirmed By These Flows

### 风险 1：把业务动作直接写成主档编辑

短期会很快，长期一定乱。

### 风险 2：把盘点当成普通单据

这样会丢掉“账面 vs 现场”的核心价值。

### 风险 3：只记录单据，不记录快照变化

这样后面查历史时还得回放逻辑，非常痛苦。

## Recommended Approach

经过这 3 条真实流程演练，当前方案可以认为已经通过了第一轮业务可表达性验证。

也就是说：

1. 表结构站得住
2. 状态模型站得住
3. 单据模型站得住
4. 盘点模型站得住

下一步就可以从“纯方案梳理”进入“正式落库与实现准备”。
