## Problem Statement

现在我们已经有了：

1. 产品方案
2. 实施方案
3. SQL 设计草案
4. 菜单与权限设计

下一步如果直接开写代码，最容易踩的坑不是“不会写”，而是“顺序错了”。

典型问题会是：

1. 先去写复杂单据流程，结果基础 CRUD 还没立住。
2. 先做前端大页面，结果后端接口分组还没定。
3. 把本来应该先写成标准 CRUD 骨架的模块和必须按流程建模的模块混在一起，导致后面返工。

所以这一版的目标，是把“接口怎么分组、哪些模块适合先写标准 CRUD 骨架、哪些必须手写流程逻辑、前后端怎么切开发节奏”先定下来。

## Constraints

1. 技术栈仍以 `RuoYi-Vue + art-design-pro` 为前提。
2. 第一阶段优先做能跑通闭环的 MVP，不追求一次性把所有页面都做精。
3. 当前实施前提是不使用代码生成器，所有模块都按手写方式落地。
4. 前后端分工要尽量围绕“接口成熟度”组织，而不是围绕文件数量组织。

## Premises

1. `asset_category`、`asset_location`、`asset_info` 适合先写标准 CRUD 骨架。
2. `asset_operate_order`、`asset_inventory_task`、`asset_event_log` 不适合按纯 CRUD 思路实现，必须手写核心逻辑。
3. 看板接口属于聚合接口，不应混入 CRUD Controller。
4. 一期目标不是做满所有 API，而是先支持 3 条关键业务流程：
- 入库并直接领用
- 跨部门调拨
- 盘点发现位置不符

## Interface Grouping Strategy

建议按“接口性质”而不是“表数量”分 4 组。

### Group A: 主数据 CRUD 接口

这组接口的特点是：

1. 数据结构稳定
2. 流程简单
3. 适合先手写成标准 CRUD 骨架

包含：

1. 分类管理
- `/asset/category/*`

2. 存放位置
- `/asset/location/*`

3. 资产台账基础信息
- `/asset/info/*`

建议优先手写的标准 CRUD 接口：

1. list
2. detail
3. add
4. edit
5. remove
6. export

### Group B: 业务单据流程接口

这组接口的特点是：

1. 状态流转复杂
2. 涉及事务
3. 会修改资产主档
4. 需要写事件流水

包含：

1. 通用业务单据
- `/asset/order/*`

核心动作：

1. 草稿保存
2. 提交审批
3. 审批通过/驳回
4. 执行完成
5. 取消

这组必须按流程手写，不能当成普通 CRUD。

### Group C: 盘点任务接口

这组接口的特点是：

1. 有任务头和任务明细
2. 有批量处理差异动作
3. 需要对比账面与现场

包含：

1. 盘点任务
- `/asset/inventory/*`

这组也建议手写。

### Group D: 聚合与辅助接口

这组接口不直接对应某一张表，而是服务页面体验。

包含：

1. 看板接口
- `/asset/dashboard/*`

2. 附件接口
- `/asset/attachment/*`

3. 资产流水接口
- `/asset/event/*`

特点：

1. 读多写少
2. 以聚合查询为主
3. 不适合按代码生成式 CRUD 思路处理

## Recommended Backend Package Split

为了避免后面 Controller 和 Service 越写越乱，建议按接口分组来拆包。

### Controller

建议：

1. `AssetCategoryController`
2. `AssetLocationController`
3. `AssetInfoController`
4. `AssetOrderController`
5. `AssetInventoryController`
6. `AssetEventController`
7. `AssetDashboardController`
8. `AssetAttachmentController`

### Service

建议分成两层：

1. 基础服务
- `AssetCategoryService`
- `AssetLocationService`
- `AssetInfoService`

2. 流程服务
- `AssetOrderService`
- `AssetInventoryService`
- `AssetEventService`
- `AssetDashboardService`

如果后面复杂度上升，可以再抽：

1. `AssetStatusFlowService`
2. `AssetPermissionService`
3. `AssetQueryService`

但一期先不要过度拆。

## What To Handwrite First

### 推荐先手写标准 CRUD 骨架

1. 分类管理
2. 存放位置
3. 资产台账基础 CRUD

原因：

1. 这三块是稳定主数据
2. 结构清晰
3. 列表、表单、导出、树形查询都适合先手写统一风格

### 推荐手写

1. 业务单据
2. 盘点任务
3. 资产流水
4. 看板接口
5. 附件接口

原因：

1. 有业务状态流转
2. 有跨表事务
3. 有聚合查询或明细组合
4. 如果先偷懒写成伪 CRUD，后面反而更难改

## Development Phases

### Phase 0: 初始化阶段

目标：
把后面开发的地基铺平。

内容：

1. 新增业务字典
2. 准备菜单与权限设计稿
3. 整理 SQL 设计草案
4. 确认 3 条关键业务流程

产出：

1. 文档稳定
2. 开发边界稳定

### Phase 1: 主数据可用

目标：
先把“资产能被录入、维护、查询”做起来。

后端：

1. 建 `asset_category`
2. 建 `asset_location`
3. 建 `asset_info`
4. 生成对应 CRUD

前端：

1. 分类管理页
2. 位置管理页
3. 资产台账页基础版

验证标准：

1. 能新增分类
2. 能新增位置
3. 能新增资产
4. 能按状态、分类、部门查询资产

### Phase 2: 单据闭环可用

目标：
让资产真正流转起来。

后端：

1. 建 `asset_operate_order`
2. 建 `asset_operate_order_item`
3. 手写单据流程接口
4. 写资产状态回写逻辑
5. 写事件流水落库逻辑

前端：

1. 业务单据页
2. 单据详情抽屉/详情页
3. 提交、审批、完成、取消动作

验证标准：

1. 入库并领用流程可跑通
2. 调拨流程可跑通
3. 资产主档状态和归属会随单据变化
4. 事件流水可追溯

### Phase 3: 盘点可用

目标：
让系统具备“账实校验”能力。

后端：

1. 建 `asset_inventory_task`
2. 建 `asset_inventory_task_item`
3. 手写盘点任务接口
4. 支持差异处理

前端：

1. 盘点任务页
2. 盘点明细页
3. 差异处理交互

验证标准：

1. 能发起盘点任务
2. 能提交盘点结果
3. 能看到差异
4. 能完成差异处理

### Phase 4: 看板与增强

目标：
让系统从“能用”走向“可管理”。

后端：

1. 看板聚合接口
2. 流水导出
3. 附件接口
4. 批量导入

前端：

1. 资产看板页
2. 附件上传入口
3. 导入导出能力

验证标准：

1. 管理层能看到关键指标
2. 管理员能导入导出
3. 单据和资产能挂附件

## Sprint-Like Delivery Suggestion

如果你想按比较实际的节奏推进，我建议这样切：

### 第 1 批

1. 分类管理
2. 位置管理
3. 资产台账基础 CRUD

这是最适合先做的一批，因为：

1. 风险低
2. 能快速形成页面
3. 能验证若依生成器和前端接线方式

### 第 2 批

1. 通用业务单据表
2. 单据提交流程
3. 单据完成逻辑
4. 资产事件流水

这是系统真正的业务核心。

### 第 3 批

1. 盘点任务
2. 差异处理
3. 看板聚合

这是“从台账系统变成管理系统”的关键一步。

## API Stability Plan

### 哪些接口应该尽早定型

1. `/asset/category/*`
2. `/asset/location/*`
3. `/asset/info/*`

因为这些接口一旦前后端接上，就不太希望频繁变更。

### 哪些接口允许先粗后细

1. `/asset/order/*`
2. `/asset/inventory/*`
3. `/asset/dashboard/*`

因为这些更依赖真实业务细节，先跑通，再优化字段和返回结构。

## Suggested Testing Focus

### 第一阶段重点测

1. 资产台账新增与编辑
2. 分类树和位置树
3. 资产列表筛选

### 第二阶段重点测

1. 单据提交后资产状态是否正确
2. 审批驳回是否不会污染主档
3. 完成单据后事件流水是否写入

### 第三阶段重点测

1. 盘点差异识别是否准确
2. 差异处理后主档是否更新
3. 盘点任务统计是否正确

## Risks

### 风险 1：把主数据和流程一起开工

这样最容易出现：

1. 页面很多
2. 但每个都不完整

建议：

1. 主数据先独立跑通
2. 再进入流程开发

### 风险 2：过度依赖代码生成器

如果把单据流程也交给生成器，会得到一堆看上去像样、实际不好维护的代码。

建议：

1. 生成器只做主数据
2. 单据和盘点必须手写

### 风险 3：前端过早做复杂交互

一期不建议先做太重的多页签、复杂向导式单据。

建议：

1. 先用列表 + 抽屉/弹窗
2. 跑通流程优先

## Recommended Approach

最稳的开发路径不是“先把所有表建完”，而是：

1. 先立主数据
2. 再立单据闭环
3. 再立盘点
4. 最后补看板和增强功能

这个顺序的好处是：

1. 每一阶段都有可演示成果
2. 每一阶段都能验证上一阶段设计是否合理
3. 出问题时更容易定位是模型问题还是流程问题

## Open Questions

1. 前端业务单据页是否只做统一入口，还是要为常用单据做快捷入口？
2. 盘点任务是否一期就做明细页，还是先用简单列表 + 弹窗？
3. 看板首页是否优先服务资产管理员，还是同时兼顾部门主管和管理层？
