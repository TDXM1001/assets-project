# 资产单据工作台页面优先工作计划与规范

> 适用范围：资产管理系统的单据新增、编辑、审批、完成等页面化流程。
>
> 目标：把 `/asset/order/create` 及后续同类复杂流程，统一收敛为“页面优先”的资产工作台，而不是继续围绕弹层壳层做局部修补。

## 1. 背景

当前资产系统已经具备分类、位置、台账、单据、盘点、维修、事件流水和附件等主链路能力。  
其中，单据创建页已经不再是轻量表单，而是包含：

- 单据头信息
- 流转范围
- 资产明细
- 草稿保存
- 提交与审批
- 返回列表与上下文恢复

这类页面如果继续按“弹层复用”思路处理，后续会在样式、动效、上下文、草稿和审批流上持续分叉，返工风险会越来越高。  
因此，这一轮需要先把工作台边界和执行规范定死。

## 2. 总目标

1. 将 `/asset/order/create` 明确为**页面本体**，不是弹层伪装页。
2. 让单据创建、详情、审批、完成共享同一套“工作台语义”。
3. 让页面风格、动效、区块层级向 `/asset/info/create` 对齐，保证系统整体感。
4. 让上下文、草稿、返回态都可预测、可复用、可测试。
5. 为后续新增单据类型、批量维修、多资产流转打下统一底座。

## 3. 设计规范

### 3.1 页面优先原则

- 主流程一律优先使用页面承载。
- 字段多、分区多、包含明细表或多步流转的场景，禁止继续包在大弹层里。
- 弹层只允许承载轻量动作：
  - 资产选择
  - 二次确认
  - 快速补录的短表单
  - 局部搜索

### 3.2 视觉统一原则

- 所有资产页面都要遵守同一套视觉语言：
  - 同样的背景节奏
  - 同样的 hero/标题区
  - 同样的卡片边距与圆角
  - 同样的底部 sticky 动作栏
  - 同样的轻量进入动效
- `/asset/order/create` 的样式必须向 `/asset/info/create` 靠齐，不允许形成独立“页面风格分支”。
- 统一动效建议使用同一进入动画，不允许每个页面各写一套视觉效果。

### 3.3 上下文单一入口原则

- 创建页必须通过统一上下文对象接收入口信息。
- 上下文字段至少包括：
  - `orderType`
  - `bridgeSource`
  - `bridgeKey`
  - `sourceBizType`
  - `sourceBizId`
  - `repairId`
- 上下文解析只允许有一个标准入口，禁止在页面里散落多处 `route/query/sessionStorage` 逻辑。

### 3.4 草稿隔离原则

- 草稿不能只按“单据类型”存。
- 必须按“单据类型 + 来源业务 + 来源 ID + 关联上下文”组合隔离。
- 目标是避免：
  - 不同来源的草稿互相覆盖
  - 维修入口和列表入口相互污染
  - 页面重开后恢复错草稿

### 3.5 组件复用边界原则

- 可以复用的部分：
  - 页面头部
  - 区块标题
  - 底部动作栏
  - 表单分区
  - 资产选择器
  - 草稿工具函数
- 不建议复用的部分：
  - 页面本体外壳
  - 主工作流容器
  - 以弹层为中心的外层结构

### 3.6 动态路由原则

- 保持现有动态路由体系，不回退到静态路由。
- 创建页必须仍然归属资产模块菜单树。
- 新增页面的“页面化”只改变承载形态，不改变系统路由归属。

### 3.7 SQL 与数据规范

- 只有在确实需要新字段或新结构时，才允许新增增量 SQL。
- 增量 SQL 必须满足：
  - 名称可追溯
  - 内容幂等
  - 变更目的明确
  - 执行前确认现有库里是否已经存在目标结构
- 不允许为了“看起来一致”而重复执行已有脚本。

### 3.8 文案与编码规范

- 新增代码注释必须使用中文。
- 新增文件必须以 UTF-8 保存。
- 用户可见文案必须与系统整体中文语境一致。
- 禁止在同一模块里混入不同风格的提示语和标题。

## 4. 工作计划

### 阶段 1：定义页面规范基线

目标：把工作台页面的结构、样式、动效、上下文标准先钉牢。

交付物：
- 页面结构规范
- 动态路由上下文规范
- 草稿隔离规范
- 弹层边界规范

验收标准：
- 任一资产单据页面都能按同一规则接入
- 没有新的“页面风格分支”

### 阶段 2：重构单据创建页为页面本体

目标：让 `/asset/order/create` 真正成为页面，而不是“像页面的弹层”。

交付物：
- 创建页页面骨架
- 页面头部和卡片区
- 底部动作区
- 与资产新增页一致的视觉节奏

验收标准：
- 页面打开后不依赖弹层外壳
- 视觉和 `asset/info/create` 统一
- 可独立访问、刷新、返回

### 阶段 3：统一上下文和草稿

目标：把所有入口的上下文、草稿、返回态收口到同一套实现。

交付物：
- 统一上下文解析器
- 统一草稿 key 计算
- 返回列表筛选态恢复
- 来源业务回流

验收标准：
- 从列表、维修、报废等入口进入创建页，行为一致
- 草稿不会互相串

### 阶段 4：统一单据类型配置

目标：把单据类型对应的标题、说明、标签、默认动作统一收敛。

交付物：
- 单据类型配置表
- 类型驱动的页面文案
- 类型驱动的默认标签

验收标准：
- 新增单据类型时，尽量只改配置，不改壳层结构

### 阶段 5：浏览器验收与收口

目标：用实际页面烟雾测试验证工作台体验。

验收项：
- `/asset/order/create` 是否像 `/asset/info/create` 一样是纯页面体验
- 页面是否保持系统内归属感
- 返回是否保留上下文
- 草稿是否按来源隔离
- 动效与页面节奏是否统一

## 5. Story 拆分建议

建议按下面的故事线推进：

- `STORY-E008-S001` 单据工作台页面本体定稿
- `STORY-E008-S002` 上下文回流与草稿隔离
- `STORY-E008-S003` 单据类型驱动的页面配置统一

如果后续还有扩展，可以继续拆：

- `STORY-E008-S004` 单据页面与资产新增页视觉对齐收口
- `STORY-E008-S005` 页面级 QA 与回归补强

## 6. 实施约束

### 必须做

- 页面优先，不再把主流程塞回弹层。
- 统一视觉，不允许资产模块各页面“各长各的样子”。
- 统一上下文，不允许入口逻辑散落在多个文件里。
- 统一草稿，不允许不同来源互相污染。
- 中文注释、UTF-8、增量 SQL 可追溯。

### 不能做

- 不把单据创建页再改回大弹层。
- 不为了复用外壳牺牲页面一致性。
- 不在页面里继续堆多套解析逻辑。
- 不把局部临时方案升级成长期结构。

## 7. 建议的执行方式

这条线适合继续使用：

- **gstack** 作为主力执行与 QA 入口
- **superpowers** 作为规划、分工和并行执行辅助

推荐执行顺序：

1. 先定规范
2. 再改页面本体
3. 然后统一上下文和草稿
4. 最后做浏览器验证和收口

## 8. 结论

`/asset/order/create` 不是动态路由错误，而是一个已经处在动态路由体系里的复杂工作台页面。  
这一轮最重要的不是继续围绕弹层修修补补，而是先把“页面优先”的规范定下来，再按规范推进后续实现。

只要规范先立住，后面每个新单据类型、每个新页面、每个新入口，都会更可控，返工风险也会明显下降。
