# 资产单据工作台统一壳设计草案

**日期：** 2026-03-27  
**阶段：** 设计草案  
**目标：** 让资产单据创建、详情、审批、完成沿用同一套系统壳与上下文，不再表现成“跳出系统的独立页面”。

## 1. 背景

当前资产系统已经完成了分类、位置、台账、单据、盘点、维修、事件流水、附件和看板等主链路。`asset/order/create` 这页并不是静态页，也不是系统外页面，它已经通过动态路由和菜单数据被挂进了资产系统内。

但从用户视角看，它仍然像一个“单独打开的新页面”，原因不是路由错了，而是页面产品形态还不够统一：

- 列表页、创建页、详情页之间的上下文连续性不够强。
- 创建页的头部、动作区、返回路径和列表页没有统一的工作台语义。
- 当前单据页已经从“简单表单”变成“流程型对象”，继续用普通页面的方式表达，会让用户感到割裂。

## 2. 问题定义

我们要解决的不是“是不是动态路由有问题”，而是下面这几个产品问题：

- 用户进入创建页后，是否还知道自己处在资产系统的哪个工作流里。
- 创建页、详情页、审批页、完成页是否像同一个业务对象的不同视图。
- 从列表进入创建页、从维修/报废跳入单据页、再返回列表时，能否保持上下文连续。
- 后续如果再增加单据类型，是否还能复用同一套页面结构。

## 3. 目标

1. 保留动态路由体系，不引入静态路由。
2. 让单据创建页成为“单据工作台”的一部分，而不是孤立的独立页面。
3. 让列表页、创建页、详情页、审批页共享同一套上下文参数和返回策略。
4. 让草稿、提交、审批、完成等动作在同一工作台语义下表达。
5. 为后续新增单据类型、单据模板和多资产流转打基础。

## 4. 非目标

1. 不把单据创建重新收回弹窗。
2. 不把动态路由改成静态路由。
3. 不重做整套菜单系统。
4. 不在这一轮引入复杂流程引擎。
5. 不做低代码表单平台。

## 5. 方案对比

### 方案 A：只补系统壳感

做面包屑、页面说明、返回列表保留筛选态、统一页面头部与按钮层级。

优点：
- 改动小。
- 见效快。
- 可以迅速改善“像跳出系统”的观感。

缺点：
- 本质上还是独立页面。
- 以后单据流程继续变复杂时，还是会回到碎片化表达。

### 方案 B：单据工作台统一壳

保留独立页面，但把创建、详情、审批、完成统一为一个工作台语义，页面之间通过稳定上下文流转。

优点：
- 最符合当前资产单据的复杂度。
- 最适合后续扩展更多单据类型。
- 既保留页面化能力，也保留系统归属感。

缺点：
- 设计和实现成本略高于方案 A。

### 方案 C：收回到列表页内联

把创建动作塞回列表页抽屉或内联表单，尽量不跳转。

优点：
- 视觉上最紧凑。
- 用户不离开列表页。

缺点：
- 不适合当前单据复杂度。
- 未来继续扩展时会越来越难维护。

**推荐：方案 B**

## 6. 推荐设计

### 6.1 路由策略

继续使用动态路由，不新增静态路由。单据创建页仍然作为资产菜单体系中的一个路由节点存在，但视觉与交互上要纳入“工作台”。

原则如下：

- 列表页进入创建页时，通过路由参数传递上下文。
- 创建页可以独立打开，不依赖列表页状态。
- 从维修、报废或其他来源跳转时，返回动作应该回到原始上下文。
- 同一类型的单据草稿应按上下文隔离，避免串草稿。

### 6.2 页面壳结构

单据工作台建议固定为四个区域：

1. 顶部状态区
2. 主信息区
3. 明细区
4. 底部动作区

顶部状态区负责表达：

- 单据类型
- 单据状态
- 草稿 / 提交 / 审批 / 完成标签

主信息区负责表达：

- 发起信息
- 流转范围
- 补充字段

明细区负责表达：

- 资产选择
- 行级编辑
- 批量操作

底部动作区负责表达：

- 保存草稿
- 保存并继续
- 提交并返回
- 审批
- 驳回
- 完成
- 取消

### 6.3 上下文传递

建议保留并统一以下上下文字段：

- `orderType`
- `bridgeSource`
- `bridgeKey`
- `sourceBizType`
- `sourceBizId`
- `repairId`

这些字段不只是为了跳转，更是为了：

- 草稿隔离
- 返回定位
- 来源业务回流
- 单据类型分流

### 6.4 草稿策略

草稿不应该只按“单据类型”保存，而应该按“单据类型 + 来源业务 + 来源 ID + 维修/报废关联”组合保存。这样可以避免：

- 一个来源业务覆盖另一个来源业务的草稿。
- 不同路径进入的新建页互相污染。

### 6.5 返回策略

返回列表时要尽量保留：

- 当前单据类型
- 当前筛选条件
- 来源业务状态
- 列表页原始上下文

这会让用户感觉自己始终在一个工作台里，而不是在多个页面之间跳来跳去。

## 7. 组件与职责

建议把工作台拆成职责清晰的几块：

- 工作台壳层：负责整体页面结构和统一语义。
- 单据头信息区：负责类型、状态、发起人与流转范围。
- 单据明细区：负责资产选择和行级编辑。
- 动作区：负责草稿、提交、审批、完成等动作。
- 上下文适配层：负责从列表页、维修页、报废页等入口接收参数。

这几个单元之间要保持边界清晰，避免一份页面里既管路由、又管表单、又管审批、又管草稿。

## 8. 数据流

推荐的数据流是：

1. 列表页或来源页发起跳转。
2. 创建页接收上下文并初始化草稿键。
3. 用户编辑头信息和明细。
4. 用户保存草稿或提交。
5. 后端生成单据并写入事件流水。
6. 详情页或审批页继续承接同一单据。
7. 完成或取消后回到列表页，并恢复原上下文。

这个数据流的核心不是“跳转很多次”，而是“对象始终是同一个对象”。

## 9. 错误与边界

需要显式处理这些边界：

- 直接访问创建页时没有上下文。
- 草稿版本与当前模板不一致。
- 用户权限不足时，看得到列表但进不去创建页。
- 来源业务已失效时，页面要降级成通用新建单据。
- 关闭页面后重开，草稿仍要可恢复，但只恢复同一上下文下的草稿。

## 10. 测试重点

后续进入实现时，测试重点应该是：

- 动态路由是否仍然在资产系统布局壳内。
- 创建页是否保留上下文参数。
- 返回列表是否恢复筛选状态。
- 草稿是否按上下文隔离。
- 维修、报废、通用单据进入创建页后，标题、类型和动作区是否一致。
- 直接访问创建页时是否能正常降级。

## 11. 后续故事线建议

如果这条设计方向通过，下一批 story 建议拆成：

- `STORY-E008-S001` 单据工作台统一壳
- `STORY-E008-S002` 上下文回流与草稿隔离
- `STORY-E008-S003` 单据类型驱动的页面结构统一

## 12. 结论

`asset/order/create` 不是动态路由错误，而是一个已经在动态路由体系中的独立工作页。  
当前要做的，不是把它改回静态路由，也不是把它塞回弹窗，而是把它升级成资产单据工作台的一部分。

这条线是对现有资产系统的产品增强，不是推翻重做。
