# 资产维修页面优先设计草案

> 适用范围：资产管理系统中的维修管理模块，重点覆盖新增、编辑、详情三类主流程页面。
>
> 设计目标：把 `repair` 从“列表 + 弹层/抽屉混合形态”收口为“页面优先”的标准业务工作流，并与 `asset/info/create` 的视觉与交互语言保持一致。

## 1. 背景与问题

当前维修模块已经具备列表、详情抽屉、编辑弹层、审批弹层和完工弹层等能力，业务上可以跑通，但页面形态上仍然偏向“局部弹层拼接”。

这个形态的主要问题有三类：

1. 主流程承载不稳定。新增、编辑、详情都被拆在不同浮层里，页面级上下文不完整。
2. 视觉语言不统一。维修模块如果继续保留自己的弹层主流程，很容易和资产新增页、单据工作台出现风格分叉。
3. 返工风险高。后续只要再加字段、加流程、加附件、加批量操作，弹层都会更快到达承载上限。

因此，维修模块需要明确转向“页面优先”，把主流程统一到独立页面里。

## 2. 目标

1. 将 `/asset/repair/create` 明确为页面本体，而不是弹层伪装页。
2. 将 `/asset/repair/edit/:repairId` 明确为独立编辑页。
3. 将 `/asset/repair/detail/:repairId` 明确为独立详情页。
4. 让维修模块的视觉、动效、卡片结构和底部动作栏与 `asset/info/create` 对齐。
5. 让上下文、草稿、返回态、状态机都能稳定复用，不再串数据。
6. 只保留局部动作的轻量浮层，不再保留主流程弹层。

## 3. 页面边界

### 3.1 路由结构

建议维修模块采用如下页面路由：

- `/asset/repair`
  - 列表页
  - 负责查询、筛选、状态切换、导出、入口导航
- `/asset/repair/create`
  - 新增维修单
- `/asset/repair/edit/:repairId`
  - 编辑维修单
- `/asset/repair/detail/:repairId`
  - 维修单详情

如后续审批、完工等动作需要更完整承载，可以再增加：

- `/asset/repair/approve/:repairId`
- `/asset/repair/finish/:repairId`

但在当前阶段，不建议一次性拆得过细。

### 3.2 列表页职责

列表页只保留“导航与检索”职责，不再承担主编辑流和主详情流。

列表页保留的能力：

- 搜索
- 筛选
- 状态切换
- 导出
- 新增入口
- 详情入口
- 编辑入口
- 删除、提交、完工等列表级动作

列表页不再承担：

- 主单编辑
- 主详情展示
- 维修完工主表单
- 审批主表单

### 3.3 页面级职责

#### 新增页

`/asset/repair/create` 负责：

- 新建维修单
- 录入主单信息
- 录入维修资产明细
- 草稿保存
- 提交

#### 编辑页

`/asset/repair/edit/:repairId` 负责：

- 编辑草稿或可编辑状态的维修单
- 继续补录资产明细
- 保持与新增页同一结构

#### 详情页

`/asset/repair/detail/:repairId` 负责：

- 查看维修单完整信息
- 查看资产明细、审批轨迹、完工结果、附件等
- 按状态暴露可执行动作

## 4. 视觉与交互规范

### 4.1 统一视觉基线

维修模块的页面风格必须向 [资产新增页](../../../../art-design-pro/src/views/asset/info/create/index.vue) 对齐：

- 浅色页面背景
- hero 区块
- 草稿提示条
- 卡片式内容分区
- 底部 sticky 动作栏
- 轻量进入动效

维修模块不再单独发展一套“维修风格”。

### 4.2 页面骨架统一

create/edit/detail 三类页面都应遵循同一骨架：

- 顶部 hero
  - 页面标题
  - 页面说明
  - 状态标签
- 草稿提示
  - 仅在 create/edit 页面显示
- 主体内容
  - 主单信息
  - 维修资产明细
  - 处理结果
  - 附件与补充信息
- 底部动作栏
  - 主次按钮层级统一
  - 固定在视口底部

### 4.3 新增与编辑页面规则

新增页和编辑页应共享同一套页面结构，只允许在以下方面产生差异：

- 标题文案
- 初始数据来源
- 按钮文案
- 是否只读
- 是否允许提交

不允许在以下方面出现差异：

- 页面壳层
- 卡片结构
- 间距节奏
- 背景与阴影
- 动效

### 4.4 详情页规则

详情页必须是独立页面，不再使用抽屉承载主详情。

详情页要求：

- 保留与 create/edit 相同的视觉骨架
- 以只读描述和表格为主
- 底部只显示与当前状态匹配的动作
- 返回列表时保留筛选态

### 4.5 局部浮层边界

以下能力可以继续以局部浮层存在：

- 资产选择器
- 二次确认
- 小型补充说明
- 附件选择

以下能力不再允许作为主流程载体：

- 主单编辑弹层
- 主详情抽屉
- 主审批弹层
- 主完工弹层

## 5. 路由与状态流转

### 5.1 上下文统一入口

维修页面应从统一上下文接收入口信息，至少包括：

- `sourcePage`
- `bridgeSource`
- `bridgeKey`
- `sourceBizType`
- `sourceBizId`
- `assetId`
- `repairId`

原则：

- 页面可以独立打开
- 如果有上下文，就按上下文初始化
- 如果没有上下文，就按默认新建流处理
- 页面不能依赖列表页状态才能工作

### 5.2 草稿隔离

维修草稿不能只按“单据类型”隔离，必须至少按以下组合隔离：

- 来源页面
- 来源业务类型
- 来源业务 ID
- 资产 ID
- `repairId`

这样可以避免：

- 不同入口的草稿互相覆盖
- 多资产维修和单资产维修串草稿
- create 和 edit 恢复错单

### 5.3 状态机

维修单状态建议继续保持以下集合：

- `DRAFT`
- `SUBMITTED`
- `APPROVED`
- `REJECTED`
- `FINISHED`
- `CANCELED`

页面行为应与状态绑定：

- `create`
  - 可保存草稿
  - 可提交
- `edit`
  - 仅草稿、驳回、可编辑状态允许进入
- `detail`
  - 所有状态可查看
  - 只根据状态暴露对应动作
- `list`
  - 负责状态过滤和入口导航

### 5.4 返回态

从 create / edit / detail 返回列表时，应保留：

- 原来的状态筛选
- 原来的关键字搜索
- 原来的分页

如果来源是其他业务页面，则应回到来源页面，不要强制跳回维修列表。

## 6. 组件复用与实现边界

### 6.1 可以复用的部分

以下能力可以继续复用，但只能作为局部组件：

- 资产选择器
- 二次确认弹层
- 审批/完工小动作
- 附件入口
- 只读信息块

### 6.2 不建议继续沿用的部分

以下组件不建议继续作为主流程载体：

- `repair-dialog.vue`
- `repair-detail-drawer.vue`
- `repair-approve-dialog.vue`
- `repair-finish-dialog.vue`

这些组件可以保留为局部片段，但不能再承担整张单据的主编辑流。

### 6.3 推荐的组件层次

- `repair/index.vue`
  - 列表页
- `repair/create/index.vue`
  - 创建页
- `repair/edit/index.vue`
  - 编辑页
- `repair/detail/index.vue`
  - 详情页
- `repair/modules/*`
  - 局部复用组件

## 7. SQL 与代码规范

### 7.1 SQL 规则

仅在确实需要新字段或新结构时才允许新增增量 SQL。

增量 SQL 必须满足：

- 名称可追溯
- 内容幂等
- 变更目的明确
- 执行前确认本地库是否已存在目标结构

### 7.2 代码与文档规范

- 新增代码注释必须使用中文
- 新增文件必须以 UTF-8 保存
- 用户可见文案必须与系统中文语境统一
- 同一模块内不混入多种风格的标题和提示语

## 8. 交付节奏建议

建议按以下顺序推进：

1. 先定页面边界和路由
2. 再改页面壳和视觉规范
3. 然后统一上下文、草稿和返回态
4. 最后收敛组件边界和局部浮层

这样可以尽量把返工控制在最小范围内。

## 9. 验收标准

维修模块页面优先改造完成后，应满足以下标准：

- 新增、编辑、详情都有独立页面
- 列表页不再承担主编辑流和主详情流
- 页面视觉与 `asset/info/create` 统一
- 草稿不会串
- 上下文不会串
- 返回列表时保留筛选态
- 只保留必要的局部弹层，不再保留主流程弹层

## 10. 结论

维修模块的页面化改造，核心不是“再做几个页面”，而是把资产系统的工作流表达方式统一起来。

只要这次把页面边界、视觉规则、状态流转和组件边界钉死，后续所有资产页面都会更稳定，返工风险也会明显下降。
