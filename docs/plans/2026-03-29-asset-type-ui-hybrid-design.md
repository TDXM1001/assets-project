# 资产类型前端屏蔽与动态 JSON 架构设计 (UI Hybrid Design)

## 1. 业务背景
响应 `gstack` 提出的 **“湖泊与海洋（Lake vs. Ocean）”** 原则，我们放弃了极其危险的“物理主子表拆分”路线（Boil the Ocean）。转而执行更加务实安全的路线：在绝对不破坏现有 `asset_info` 主线查询结构和其下游依赖（如维修工作台、全站资产盘点字典等）的前提下，通过智能的前端动态渲染与底层的分类 JSON 模板引擎，完美平替异构资产间的隔离诉求。

## 2. 核心架构设计

### 2.1 底层存储不破窗 (Database & Backend)
- **物理表保持原状**：`asset_info` 的 `brand`（品牌）, `model`（型号）, `specification`（规格）, `serial_no`（SN码）, `warranty_expire_date`（质保期）等硬件标配列继续保留在主表，享受最佳的 MySQL 单表模糊检索性能。
- **模板处理孤儿字段**：专门遇到诸如“不动产”的特殊高差异化属性（如“产权证号”、“房屋面积”、“土地用途”），直接依赖系统已构建强大的 `extra_fields_json` 分类模板解析器接管存储和渲染。不去污染物理主表。

### 2.2 视觉隔离层 (Frontend UI)
- 将系统已定义的资产划分为两大阵营：“硬件实体类”（FIXED_ASSET, LOW_VALUE）与“非硬件虚拟/空间类”（REAL_ESTATE）。
- 在“资产新增”（`create/index.vue`）与“资产编辑”（`info-dialog.vue`）界面启用强 UI 屏蔽墙：
  当用户在上游 `assetType` 选择了非硬件类别（如 `'REAL_ESTATE'`）时，底层触发计算属性，强行通过原生指令 `v-if="false"` 将全部的硬件物理输入框阻断/隐藏。
- 数据落盘时，这些被界面雪藏的硬件字段会被携带为原生 `null` 或未定义落入数据库，0 代码入侵，100% 不会冲撞任何现有资产 SQL 的业务判断（空列本身对于该业务是合理的静默状态）。
