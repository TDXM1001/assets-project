## 目标

这份文档把前面的数据库、菜单、角色方案继续推进到“后端应该先建什么代码骨架”的层级，重点回答三件事：

1. 资产模块在当前若依项目里应该落到哪些目录
2. 在“不使用代码生成器”的前提下，哪些模块应该先手写标准 CRUD 骨架，哪些必须先做流程骨架
3. 第一阶段后端最小可运行骨架应该包含哪些类和接口

## 项目基线

当前后端是标准若依多模块结构：

1. `RuoYi-Vue/ruoyi-admin`
2. `RuoYi-Vue/ruoyi-system`
3. `RuoYi-Vue/ruoyi-common`

结合现有目录可确认：

1. Controller 放在 `ruoyi-admin/src/main/java/com/ruoyi/web/controller/...`
2. 业务 domain / mapper / service 放在 `ruoyi-system/src/main/java/com/ruoyi/system/...`
3. Mapper XML 放在 `ruoyi-system/src/main/resources/mapper/system`
4. 基础领域对象如 `BaseEntity`、`TreeEntity` 在 `ruoyi-common`

所以资产模块最稳的落法是：

1. Controller 放 `ruoyi-admin`
2. 业务实体、Mapper、Service 放 `ruoyi-system`
3. 不要新开一个 `ruoyi-asset` 模块

当前实施前提补充：

1. 不使用若依代码生成器产出业务代码
2. 所有 Controller、Service、Mapper、XML 均按资产模块手写
3. 可以参考若依现有系统模块写法，但不依赖生成器模板

原因：

1. 一期规模还不需要额外拆 Maven 模块
2. 可以最大程度复用现有若依启动、权限、日志、分页、导出能力
3. 成本最低，后面需要拆模块时也还有余地

## 推荐目录结构

### `ruoyi-admin`

建议新增：

1. `RuoYi-Vue/ruoyi-admin/src/main/java/com/ruoyi/web/controller/asset`

目录下建议放：

1. `AssetCategoryController.java`
2. `AssetLocationController.java`
3. `AssetInfoController.java`
4. `AssetOrderController.java`
5. `AssetInventoryController.java`
6. `AssetEventController.java`
7. `AssetDashboardController.java`

说明：

1. 不建议继续塞进 `controller/system`
2. 新开 `controller/asset` 更清晰，也便于后面接口统一按 `/asset/**` 分组

### `ruoyi-system`

建议新增以下目录：

1. `RuoYi-Vue/ruoyi-system/src/main/java/com/ruoyi/system/domain/asset`
2. `RuoYi-Vue/ruoyi-system/src/main/java/com/ruoyi/system/domain/asset/bo`
3. `RuoYi-Vue/ruoyi-system/src/main/java/com/ruoyi/system/domain/asset/vo`
4. `RuoYi-Vue/ruoyi-system/src/main/java/com/ruoyi/system/mapper/asset`
5. `RuoYi-Vue/ruoyi-system/src/main/java/com/ruoyi/system/service/asset`
6. `RuoYi-Vue/ruoyi-system/src/main/java/com/ruoyi/system/service/asset/impl`

对应 XML：

1. `RuoYi-Vue/ruoyi-system/src/main/resources/mapper/asset`

说明：

1. `domain/asset` 放实体
2. `bo` 放新增、修改、提交、审批等入参对象
3. `vo` 放列表、详情、看板、树形下拉等出参对象
4. 不建议把所有资产类平铺到 `com.ruoyi.system.domain` 根目录

## 推荐分层职责

### Controller 层

负责：

1. 路由定义
2. 权限注解
3. 分页启动 `startPage()`
4. 参数校验
5. 导出调用

不建议负责：

1. 状态机流转
2. 跨表写入
3. 盘点差异计算

### Service 层

负责：

1. 业务动作编排
2. 状态流转校验
3. 跨表事务
4. 事件流水记录
5. 数据范围补充过滤

### Mapper 层

负责：

1. 单表查询
2. 列表聚合查询
3. 批量更新
4. 看板统计 SQL

## 关键服务建议

除了上面的按表 Service，还建议补 3 个偏“业务动作型”的内部服务。

### 1. `AssetFlowService`

职责：

1. 单据动作流转
2. 状态变更前后校验
3. 落 `asset_event_log`

### 2. `AssetInventoryEngineService`

职责：

1. 盘点任务生成明细
2. 计算账实差异
3. 处理位置不符、责任人不符、盘亏盘盈

### 3. `AssetPermissionService`

职责：

1. 统一补“本人资产 / 本人单据 / 本部门数据”过滤条件
2. 避免把这些判断散落到每个 Controller 里

说明：

1. 这三个服务可以一开始不单独对外暴露接口
2. 但建议在设计上预留，不要把所有逻辑都堆进 `*ServiceImpl`

## 全手写建议

建议手写时分成两类：

1. 标准 CRUD 骨架先写
2. 流程型骨架单独写

### 适合先手写标准 CRUD 骨架

1. 分类管理
2. 存放位置
3. 资产台账基础 CRUD

原因：

1. 这三块结构相对稳定
2. 手写成本可控
3. 先把目录、权限、分页、导出、VO/BO 风格统一下来，后面整个资产域代码会更整齐

### 从第一版就要按流程型模块手写

1. 业务单据
2. 盘点任务
3. 资产流水
4. 看板统计

原因：

1. 它们不是标准单表 CRUD
2. 有状态流转、跨表事务和聚合查询
3. 如果为了省事先按 CRUD 写，后面返工会更重

## 第一阶段最小骨架

## 当前不建议做的事

### 1. 不建议一开始就建非常多的 DTO / Converter

一期先保持：

1. `Entity`
2. `Bo`
3. `Vo`

三层够用。等模块稳定后再决定是否引入转换器。

### 2. 不建议把所有业务动作都写成 Controller 特殊接口

像“提交、审批、完成、处理差异”这些动作可以有动作接口，但核心逻辑一定要留在 Service。

### 3. 不建议让事件流水支持前台直接新增

事件流水应由业务动作自然产生，而不是给用户一个“新增流水”按钮。
