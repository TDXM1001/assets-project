# Design: 企业内部资产管理系统

## Problem Statement

当前项目基于 `RuoYi-Vue + art-design-pro`，数据库名已经切到 `my-assets`，但仓库内还没有真正的资产管理业务模块。基于这个上下文，我将本次目标定义为：

为企业内部建立一套可落地的资产管理系统，解决资产从采购入库、建账、领用、借用、归还、调拨、维修、盘点到报废处置的全生命周期管理问题，让“资产归谁、在哪、什么状态、值多少钱、何时该处理”都能被系统追踪。

这不是一个“只是做几张 CRUD 表”的后台，而是一个围绕责任、流程、追溯和协同的运营系统。

## What Makes This Cool

这套系统真正有价值的地方，不是把 Excel 搬到网页上，而是把资产管理里最容易失控的三件事变成可追踪事实：

1. 资产责任清楚：每一件资产都有归属部门、当前责任人、当前位置、使用状态。
2. 生命周期清楚：每一次入库、领用、归还、维修、盘点、报废都有事件记录。
3. 管理动作清楚：审批、异常、差异、逾期、待办都能被看见，而不是靠人记忆。

## Constraints

1. 当前技术基座应优先复用现有仓库能力，而不是重造后台框架。
2. 后端约束：以 `RuoYi-Vue` 的用户、角色、部门、菜单、字典、日志、代码生成器为基础。
3. 前端约束：以 `art-design-pro` 现有列表页、搜索栏、弹窗表单、树结构、权限指令和仪表盘样式为基础。
4. 第一阶段更适合聚焦“固定资产管理”，不建议一开始把低值易耗品、采购、财务折旧、供应商、合同、二维码移动端、IoT 盘点全部打包。
5. 这个系统首先是企业内部运营系统，不是对外产品，不需要面向公众的复杂增长和营销逻辑。

## Premises

以下前提是我基于当前仓库和你的提问做出的工作假设：

1. 这是一套企业内部资产管理系统，而不是二手交易、资产租赁或外部服务平台。
2. 第一阶段的核心不是“采购系统”，而是“资产台账和生命周期系统”。
3. 业务主线应围绕固定资产展开，最小闭环是：建账 -> 领用 -> 归还/调拨 -> 盘点 -> 维修/报废。
4. 权限模型以部门和角色为主，审批链先做规则化配置，不在第一阶段上 BPM 引擎。
5. 系统的第一目标是“账实一致、责任明确、流程可追溯”，而不是追求一次性覆盖所有管理想象。

## Approaches Considered

### Approach A: 台账优先的快速上线版

Summary:
先把资产基础信息、分类、领用归还、盘点、报废几个核心列表做出来，重点解决“能登记、能查询、能导出”。

Effort: M
Risk: Low

Pros:
- 最快落地，适合当前仓库快速起业务模块。
- 大量复用若依代码生成器和现有通用 CRUD 页面模式。
- 便于先让业务方看到系统雏形。

Cons:
- 业务动作容易散落成多张表，后期追踪链路会比较碎。
- 审批、异常、维修、变更历史可能很快补不动。
- 看似快，但二期很容易返工数据模型。

Reuses:
- `RuoYi-Vue` 的代码生成器、权限、日志、字典、部门树。
- `art-design-pro` 的表格页、搜索栏、抽屉/弹窗表单。

### Approach B: 生命周期驱动的完整领域版

Summary:
从一开始就以“资产主档 + 资产事件流水 + 业务单据”建模，每个动作都是一个明确事件，强调可追溯性和后续扩展性。

Effort: L
Risk: Med

Pros:
- 结构最清晰，后期扩展维修、折旧、借用、二维码、审计都顺。
- 资产状态变化有天然流水，账实差异更容易定位。
- 数据分析和审计追责能力最好。

Cons:
- 第一阶段设计成本更高，对字段、状态机和单据关系要求更严谨。
- 业务方如果只是想“先能用”，会觉得前期定义偏重。
- 若一次性做太满，容易把湖煮成海。

Reuses:
- 若依权限、组织、日志仍然可用，但生成器产物需要更多人工收敛。

### Approach C: 混合式分阶段方案

Summary:
第一阶段按“资产主档 + 核心业务单据 + 统一变更流水”落地，不做重 BPM，但把未来可扩展的骨架先立住。业务表现层仍然保持若依式后台管理体验。

Effort: M
Risk: Low-Med

Pros:
- 兼顾上线速度和长期可扩展性。
- 不会为了快而牺牲后续维修、盘点、审计扩展空间。
- 最适合当前仓库这种已有通用管理后台底座的项目。

Cons:
- 比纯 CRUD 版多一些前期建模工作。
- 需要团队先接受“单据 + 流水”而不是“每个按钮直接改状态”的思路。

Reuses:
- 若依用户、角色、部门、字典、菜单、日志、导入导出、代码生成器。
- art-design-pro 的系统管理页面模式和控制台首页能力。

## Recommended Approach

推荐选择 Approach C，因为它最适合你当前仓库的现实条件：

1. 你已经有成熟后台底座，不需要为登录、权限、菜单、日志、基础字典重开一遍工程。
2. 资产管理天然是生命周期业务，如果只做列表 CRUD，很快会碰到“状态谁改的、为什么变了、原来在哪”的追溯问题。
3. 混合式方案能在第一阶段先把最有业务价值的闭环跑通，同时给二期留足空间。

### 系统要做什么

建议把系统目标拆成五件事：

1. 建立统一资产台账
每一件固定资产都必须有唯一编码、分类、品牌型号、序列号、采购信息、所属组织、责任人、使用地点、当前状态。

2. 管理资产生命周期
资产不是一条静态记录，而是一系列动作：入库、领用、借用、归还、调拨、维修、盘点、报废。

3. 建立责任链和审批链
关键动作必须落到具体人，并对高风险动作设置审批，例如跨部门调拨、报废、维修费用超阈值。

4. 支撑盘点与异常处理
系统要能记录“账上有、现场没有”“现场有、账上没有”“责任人与位置不一致”等异常，并跟踪处理结果。

5. 提供管理视图
要让资产管理员、部门主管和管理层分别看到自己关心的待办、风险和统计。

### 推荐模块拆分

建议第一阶段按下面 8 个业务模块落地：

1. 资产分类管理
- 资产大类、小类、折旧年限、是否需要序列号、是否允许借用、是否需要盘点。

2. 资产主档管理
- 资产编码、名称、分类、规格型号、品牌、SN、采购日期、入账日期、原值、所属部门、责任人、位置、状态、备注、附件。

3. 入库建账管理
- 支持手工建账和批量导入。
- 形成“入库单/建账单”，落资产初始状态和归属。

4. 领用/借用/归还/调拨管理
- 用业务单据驱动状态变化。
- 单据完成后自动更新资产当前责任人、部门、位置、状态。

5. 维修管理
- 记录报修、送修、维修结果、费用、停用时长、是否返修。

6. 盘点管理
- 按部门/地点/分类发起盘点任务，记录盘盈、盘亏、位置不符、责任人不符。

7. 报废处置管理
- 记录报废申请、审批、处置方式、残值、处置日期和附件。

8. 统计与预警看板
- 总资产数、在用数、闲置数、维修中、待盘点、即将到期保修、待审批、异常资产数。

### 角色设计

建议第一阶段至少有以下角色：

1. 系统管理员
- 配置菜单、角色、字典、组织。

2. 资产管理员
- 维护台账，处理入库、调拨、维修、报废、盘点。

3. 部门主管
- 审批本部门领用、归还确认、调拨确认、盘点异常确认。

4. 资产使用人
- 提交领用、借用、归还、报修申请，查看自己名下资产。

5. 财务/审计角色
- 查看价值、报废、处置、盘点差异和审计流水，不一定参与所有操作。

### 权限设计

权限分三层最稳：

1. 功能权限
- 哪些菜单、按钮、接口可访问。

2. 数据权限
- 是否只能看本部门、可看本部门及子部门、还是全公司。

3. 业务动作权限
- 谁能发起、谁能审批、谁能确认、谁能作废。

建议复用若依的数据权限和部门树模型，不要自己重造一套组织权限系统。

### 核心数据对象

建议用“主档 + 单据 + 流水”的骨架：

1. 资产主档 `asset_info`
- 一件资产当前的“最新状态快照”。

2. 资产分类 `asset_category`
- 决定资产规则和统计口径。

3. 业务单据
- 如 `asset_inbound_order`
- `asset_transfer_order`
- `asset_borrow_order`
- `asset_return_order`
- `asset_repair_order`
- `asset_inventory_task`
- `asset_disposal_order`

4. 资产流水 `asset_event_log`
- 记录每次状态变化前后值、操作人、时间、来源单据、说明。

5. 附件表/标签表
- 保存图片、发票、验收单、报废证明、二维码标签等。

### 业务状态设计

资产主状态建议控制在少量稳定状态，避免失控：

1. 草稿
2. 闲置
3. 在用
4. 借出
5. 维修中
6. 盘点中
7. 待报废审批
8. 已报废

单据状态单独维护，不要把审批状态和资产状态混在一起。

## Business Flows

### 1. 资产入库建账流程

1. 资产管理员创建入库单或批量导入资产。
2. 系统校验分类、编码规则、必填字段、序列号唯一性。
3. 入库完成后生成资产主档，初始状态为“闲置”或直接指派为“在用”。
4. 系统写入资产事件流水。

### 2. 资产领用流程

1. 使用人发起领用申请，选择资产或由管理员分配。
2. 部门主管审批。
3. 资产管理员确认发放。
4. 系统更新资产责任人、所属部门、位置、状态为“在用”。
5. 系统写入领用流水和操作日志。

### 3. 借用与归还流程

1. 使用人发起借用申请，填写借用时长和用途。
2. 审批通过后资产状态改为“借出”。
3. 归还时由使用人发起或管理员登记归还。
4. 管理员验收资产状态与完整性。
5. 系统根据结果改回“闲置/在用/维修中”。

### 4. 调拨流程

1. 发起调拨申请，指定来源部门、目标部门、目标责任人和目标位置。
2. 来源部门确认资产可移交。
3. 目标部门确认接收。
4. 资产管理员最终执行调拨。
5. 系统更新所属部门、责任人、位置，并记录完整调拨轨迹。

### 5. 维修流程

1. 使用人或管理员提交报修。
2. 资产状态改为“维修中”。
3. 记录故障描述、送修方式、费用、外部供应商、完成时间。
4. 维修完成后选择“恢复在用”“转闲置”或“建议报废”。
5. 系统记录费用和维修历史。

### 6. 盘点流程

1. 资产管理员按部门/地点发起盘点任务。
2. 盘点执行人逐项确认资产在场情况、位置、责任人、状态。
3. 系统生成盘点差异清单。
4. 部门主管和资产管理员共同处理差异。
5. 差异处理完成后，系统形成盘点报告并写入流水。

### 7. 报废流程

1. 发起报废申请，填写原因、现状、残值、附件。
2. 部门主管或财务审批。
3. 资产管理员执行报废。
4. 系统把状态改为“已报废”，禁止继续领用流转。
5. 沉淀完整处置记录，供审计与财务复核。

## 页面设计建议

### 1. 首页看板

建议首页不是传统系统首页，而是“资产运营看板”：

1. 顶部指标卡
- 资产总数
- 在用资产
- 闲置资产
- 维修中
- 待审批
- 本月盘点异常

2. 中部工作区
- 我的待办
- 待审批单据
- 逾期未归还
- 即将到期保修

3. 底部分析区
- 按分类统计
- 按部门统计
- 资产状态分布
- 近 30 天资产事件趋势

### 2. 资产台账页

要像“经营控制台”，不是普通表：

1. 左侧分类/部门树筛选。
2. 中间资产列表，支持状态、责任人、位置、时间筛选。
3. 右侧详情抽屉，展示资产基础信息、当前责任关系、最近流水、附件。

### 3. 单据页

领用、归还、调拨、维修、报废建议统一交互心智：

1. 顶部筛选区。
2. 中间单据列表。
3. 详情页按“基本信息 / 流程信息 / 资产明细 / 操作记录 / 附件”分区。

## Open Questions

1. 第一阶段是否只做固定资产，还是同时覆盖低值易耗品？
2. 审批流是按部门固定配置，还是按资产类别配置？
3. 是否要求财务价值口径和实物管理口径同时在线？
4. 资产编码规则由系统生成，还是允许沿用历史编码？
5. 盘点是全盘为主，还是按部门/地点滚动盘点？
6. 是否需要一开始就做二维码标签和扫码盘点？

## Success Criteria

如果第一阶段做对了，至少要满足下面结果：

1. 任一资产可以在系统中查到唯一责任人、所属部门、当前位置和当前状态。
2. 领用、归还、调拨、报废都能形成完整单据和流水。
3. 资产管理员不再靠 Excel 人工拼接资产去向。
4. 盘点时能快速识别账实差异，并形成待处理闭环。
5. 管理层可以看到待审批、闲置、维修、异常和报废统计。

## Next Steps

1. 先确认第一阶段只做固定资产，不把范围扩散到采购、耗材、财务全链路。
2. 先画出 8 张核心表和它们之间的关系。
3. 先实现资产分类、资产主档、领用单、调拨单、盘点任务、报废单和资产流水。
4. 在前端先做“资产台账页”和“看板页”两个代表性页面，验证信息架构。
5. 再把审批、维修、二维码等增强能力逐步接入。

## The Assignment

找一位真正会管资产的人，拿他们当前正在维护的 Excel 台账或纸面流程做一次字段对账。目标不是问“你想要什么功能”，而是把下面三样真实资料拿到手：

1. 他们现在的资产字段清单。
2. 最近 1 个月真实发生过的 5 条业务记录。
3. 他们最痛的 3 个异常场景。

没有这一步，系统设计很容易看起来完整，但上线就漏关键字段和关键节点。

## What I noticed about how you think

1. 你不是只问“做哪些页面”，你先问的是“这个系统要做什么”。
2. 你把“怎么设计”和“业务之间的流程是怎么样的”放在一起，说明你关心的不只是界面，而是系统背后的业务结构。
3. 你要的是一套能落地的整体认识，而不是零散功能点清单。
