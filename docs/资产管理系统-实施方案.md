## Problem Statement

上一版文档已经确定方向：当前项目基于 `RuoYi-Vue + art-design-pro`，适合先做企业内部固定资产管理系统，目标是建立统一台账、生命周期流程、责任追踪和管理看板。

这一版继续往前走一步，不再停留在“系统应该做什么”，而是落到可实施粒度：

1. 表结构怎么拆。
2. 菜单和权限怎么挂到若依体系里。
3. 接口怎么按若依风格设计，方便全手写落地和后续扩展。

## Constraints

1. 优先复用 `sys_user`、`sys_dept`、`sys_role`、`sys_menu`、`sys_dict_*`、`sys_oper_log` 等现有能力。
2. 第一阶段只覆盖固定资产，不扩展采购、供应商、合同、财务折旧全链路。
3. 审批先做“状态流转 + 审批人/审批意见”模式，不接入 BPM 引擎。
4. 数据模型要兼顾“能快速开发”和“后续可扩展”，因此采用“资产主档 + 通用业务单据 + 盘点任务 + 事件流水”的混合式结构。

## Premises

1. 资产台账是核心事实表，所有业务动作最终都作用到 `asset_info`。
2. 领用、借用、归还、调拨、入库这类动作可先统一为“通用业务单据”，用 `order_type` 区分。
3. 盘点因为结构和流程不同，单独成任务表和明细表。
4. 报废在第一期可先纳入通用业务单据，不单独再起一套处置表；如果二期涉及财务残值和处置收入，再拆表。
5. 所有状态变化必须落 `asset_event_log`，否则后续审计和盘点差异追溯会失真。

## Approaches Considered

### Approach A: 每类业务一个独立单据表

Summary:
入库单、领用单、借用单、归还单、调拨单、报废单全部各自独立。

Effort: L
Risk: Low

Pros:
- 业务语义最直观。
- 后台页面含义清楚，业务方容易理解。

Cons:
- 第一阶段表数量和接口数量膨胀很快。
- 代码生成之后仍会有大量重复逻辑。

### Approach B: 通用业务单据 + 特殊业务拆表

Summary:
把入库、领用、借用、归还、调拨、报废统一成通用业务单据，盘点独立，维修二期再拆。

Effort: M
Risk: Low-Med

Pros:
- 最适合当前仓库快速落地。
- 表结构和接口数量更可控。
- 后续如果某类业务变复杂，再从通用单据中拆出独立表也不晚。

Cons:
- 需要提前定义清楚 `order_type`、`order_status` 和明细字段边界。

## Recommended Approach

推荐采用 Approach B。原因很简单：你现在需要的是一套能在若依体系里稳稳落下来的实施方案，而不是一开始就把所有业务表拆爆。

第一阶段推荐表分三层：

1. 主数据层
- `asset_category`
- `asset_location`
- `asset_info`

2. 业务单据层
- `asset_operate_order`
- `asset_operate_order_item`
- `asset_inventory_task`
- `asset_inventory_task_item`

3. 追踪支撑层
- `asset_event_log`
- `asset_attachment`

## Detailed Table Design

### 设计约定

1. 所有业务表默认继承若依常用审计字段：
- `create_by`
- `create_time`
- `update_by`
- `update_time`
- `remark`

2. 有逻辑删除需求的表统一带：
- `del_flag char(1) default '0'`

3. 状态类字段统一使用字典管理，不把中文硬编码进前端。

4. 用户、部门、角色不重复建表，直接关联：
- `sys_user.user_id`
- `sys_dept.dept_id`

| asset_level | varchar(20) | 分类层级，可选 |

| borrowable_flag | char(1) | 是否允许借用 |
| inventory_required_flag | char(1) | 是否纳入盘点 |

| del_flag | char(1) | 删除标志 |

### 3. 资产主档表 `asset_info`

用途：
记录每件资产当前时刻的最新快照，是整套系统的核心表。

核心字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| asset_id | bigint | 主键 |
| asset_code | varchar(64) | 资产编码，唯一 |
| asset_name | varchar(200) | 资产名称 |
| category_id | bigint | 分类 ID |
| brand | varchar(100) | 品牌 |
| model | varchar(100) | 型号 |
| specification | varchar(255) | 规格描述 |
| serial_no | varchar(100) | 序列号 |
| asset_status | varchar(20) | 当前资产状态 |
| asset_source | varchar(20) | 来源，采购/调拨/手工建账 |
| use_org_dept_id | bigint | 使用归属部门 |
| manage_dept_id | bigint | 管理归属部门 |
| current_user_id | bigint | 当前责任人 |
| current_location_id | bigint | 当前存放位置 |
| purchase_date | date | 采购日期 |
| inbound_date | date | 入库日期 |
| start_use_date | date | 启用日期 |
| original_value | decimal(18,2) | 原值 |
| residual_value | decimal(18,2) | 残值 |
| warranty_expire_date | date | 保修截止日期 |
| supplier_name | varchar(200) | 供应商名称，先文本保留 |
| qr_code | varchar(200) | 标签码/二维码值 |
| version_no | int | 乐观锁版本号 |
| status | char(1) | 记录状态 |

索引建议：

1. `uk_asset_info_code(asset_code)`
2. `uk_asset_info_serial(serial_no)`，若序列号允许为空则做普通索引
3. `idx_asset_info_category(category_id)`
4. `idx_asset_info_status(asset_status)`
5. `idx_asset_info_dept(use_org_dept_id)`
6. `idx_asset_info_user(current_user_id)`
7. `idx_asset_info_location(current_location_id)`

说明：

1. `asset_status` 只表示资产当前真实状态，不混入审批状态。
2. 业务动作引起的状态变化必须通过业务单据执行，不建议前端直接改 `asset_info`。

用途：
统一承载入库、领用、借用、归还、调拨、报废这类单据头信息。

核心字段：

| 字段 | 类型 | 说明 |
|---|---|---|

| order_no | varchar(64) | 单号，唯一 |
| order_type | varchar(20) | INBOUND / ASSIGN / BORROW / RETURN / TRANSFER / DISPOSAL |
| order_status | varchar(20) | DRAFT / SUBMITTED / APPROVING / APPROVED / REJECTED / DONE / CANCELED |
| biz_date | datetime | 业务时间 |
| apply_user_id | bigint | 发起人 |
| apply_dept_id | bigint | 发起部门 |
| approve_user_id | bigint | 审批人 |
| approve_time | datetime | 审批时间 |
| approve_result | varchar(20) | 审批结果 |
| from_dept_id | bigint | 来源部门 |
| to_dept_id | bigint | 目标部门 |
| from_user_id | bigint | 来源责任人 |
| to_user_id | bigint | 目标责任人 |
| from_location_id | bigint | 来源位置 |
| to_location_id | bigint | 目标位置 |

索引建议：

2. 如果二期某类业务复杂度明显升高，比如维修或报废处置财务化，再拆成专表。

## API List

### 设计原则

1. 路径风格保持若依风格：
- 列表：`GET /module/entity/list`
- 详情：`GET /module/entity/{id}`
- 新增：`POST /module/entity`
- 修改：`PUT /module/entity`
- 删除：`DELETE /module/entity/{id}`

2. 特殊业务动作使用明确动作接口，不把所有动作塞进更新接口。

### 1. 分类管理接口

Base Path:
`/asset/category`

接口清单：

1. `GET /asset/category/list`
- 查询分类树或列表

2. `GET /asset/category/{categoryId}`
- 查询分类详情

3. `POST /asset/category`
- 新增分类

4. `PUT /asset/category`
- 修改分类

5. `DELETE /asset/category/{categoryId}`
- 删除分类

6. `GET /asset/category/tree`
- 获取树选择器数据

7. `POST /asset/category/export`
- 导出分类

### 2. 位置管理接口

Base Path:
`/asset/location`

接口清单：

1. `GET /asset/location/list`
2. `GET /asset/location/{locationId}`
3. `POST /asset/location`
4. `PUT /asset/location`
5. `DELETE /asset/location/{locationId}`
6. `GET /asset/location/tree`
7. `POST /asset/location/export`

### 3. 资产台账接口

Base Path:
`/asset/info`

接口清单：

1. `GET /asset/info/list`
