## 目标

这份文档把前面“菜单与权限设计”推进到接近可执行的 SQL 层级，重点解决三件事：

1. `资产管理` 这一组菜单怎样初始化到 `sys_menu`
2. 页面菜单和按钮权限怎样一一落库
3. 后面转成正式 `sql/*.sql` 文件时，应该采用什么写法更稳

## 基线

当前项目里已经有若依原生菜单结构，参考文件：

1. `RuoYi-Vue/sql/ry_20250522.sql`
2. `RuoYi-Vue/ruoyi-generator/src/main/resources/vm/sql/sql.vm`

从这两处可以确认：

1. `sys_menu` 当前字段包含 `query`、`route_name`、`is_frame`、`is_cache`、`menu_type`、`perms` 等列
2. 若依默认 SQL 常见写法是直接 `insert into sys_menu values(...)`
3. 代码生成器写法更适合我们当前阶段，因为它使用显式列名，后续字段微调时更安全

## 当前建议

### 建议 1：正式初始化脚本用“显式列名插入”

不建议继续沿用：

```sql
insert into sys_menu values(...)
```

更建议统一写成：

```sql
insert into sys_menu (
  menu_name, parent_id, order_num, path, component, query, route_name,
  is_frame, is_cache, menu_type, visible, status, perms, icon,
  create_by, create_time, update_by, update_time, remark
) values (...);
```

原因很简单：

1. 可读性更高
2. 后续数据库字段顺序变化时不容易出错
3. 更贴合你后面自己维护模块初始化脚本的场景

### 建议 2：初始化脚本尽量可重复执行

最稳的方式不是写死一批 `menu_id`，而是：

1. 先 `insert ... select ... where not exists`
2. 再通过 `select menu_id into 变量` 取回刚插入或已存在的菜单 ID

这样做的好处是：

1. 不容易和已有菜单 ID 冲突
2. 开发环境反复导入更安全
3. 后续拆分脚本时不依赖固定号段

### 建议 3：先只落 `sys_menu`

这一版先只处理：

1. 目录菜单
2. 页面菜单
3. 按钮权限

`sys_role_menu` 授权建议放在下一份草案里单独处理。因为菜单初始化和角色授权虽然相关，但不是同一层问题。

## 推荐菜单结构

本轮仍然沿用前一份文档里的收敛方案：

1. 一级目录：`资产管理`
2. 二级页面：
- `资产看板`
- `分类管理`
- `存放位置`
- `资产台账`
- `业务单据`
- `盘点任务`
- `资产流水`

### 1. 初始化一级目录

```sql
insert into sys_menu (
  menu_name, parent_id, order_num, path, component, query, route_name,
  is_frame, is_cache, menu_type, visible, status, perms, icon,
  create_by, create_time, update_by, update_time, remark
)
select
  '资产管理', 0, 20, 'asset', null, '', '',
  1, 0, 'M', '0', '0', '', 'education',
  'admin', sysdate(), '', null, '资产管理目录'
where not exists (
  select 1 from sys_menu where parent_id = 0 and path = 'asset'
);

select @asset_root_id := menu_id
from sys_menu
where parent_id = 0 and path = 'asset'
limit 1;
```

## 脚本执行顺序

真正落成 SQL 文件时，建议顺序固定成下面这样：

1. 先插一级目录 `资产管理`
2. 再插二级页面菜单
3. 每插完一个页面菜单，立即回查出对应 `menu_id`
4. 再插这个页面下的按钮权限

不要反过来先批量插按钮，再回头找父菜单。这样脚本更难维护，也更容易漏权限。

## 当前不建议做的事

### 1. 不建议现在就硬编码一长串固定 `menu_id`

如果后面环境里已经有自定义菜单，固定 ID 很容易冲突。当前阶段先追求“可重复执行”和“容易转正式脚本”，比追求号段整齐更重要。

### 2. 不建议把“领用单、调拨单、报废单”拆成三个页面菜单

一期仍然建议统一收敛在 `业务单据` 菜单下，页面内部按 `order_type` 做切换。这样可以少一层菜单膨胀，也更利于复用接口和状态机。

### 3. 不建议把数据权限写进菜单 SQL

菜单只解决“能不能访问这个页面或按钮”，不解决“能看到哪些部门数据”。数据范围仍然应该由角色和后端查询条件控制。
