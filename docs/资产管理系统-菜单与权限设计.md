## Problem Statement

前面两份文档已经把业务边界、表结构和 SQL 设计草案说明清楚了。接下来最容易失控的部分，其实不是表，而是菜单和权限。

如果菜单挂得乱、按钮权限命名不统一、角色边界不清晰，后面会出现三类问题：

1. 前端菜单和后端接口权限标识对不上。
2. 资产管理员、部门主管、普通使用人都能看到不该看的操作。
3. 一旦开始初始化 `sys_menu` 和角色授权，就会因为权限粒度不一致而返工。

这一版文档的目标，就是把“菜单怎么挂、权限怎么命名、角色怎么映射、初始化顺序怎么排”先定下来。

## Constraints

1. 权限体系必须复用若依现有 `sys_menu + sys_role + sys_role_menu` 模型。
2. 前端权限和后端权限标识必须完全一致。
3. 第一阶段先做稳定的业务权限，不引入复杂的行级权限规则引擎。
4. 数据范围控制优先复用若依角色中的数据权限模型，而不是另外造一套。

## Premises

1. 菜单权限标识沿用若依风格：`模块:资源:动作`。
2. 目录和页面菜单本质上也是 `sys_menu` 数据，只是 `menu_type` 不同。
3. 资产管理模块既要支持后台管理人员，也要支持普通资产使用人。
4. 一期角色不宜过多，先把最核心的 4 到 5 类角色定义清楚。

## Permission Design Principles

### 原则 1：菜单权限名必须稳定

一旦命名进入前端 `v-auth`、后端 `@PreAuthorize`、角色授权和测试用例，就很难再随意改。

因此命名要做到：

1. 模块固定：`asset`
2. 资源明确：`category / location / info / order / inventory / event / repair / dashboard`
3. 动作有限：`list / query / add / edit / remove / export / import / submit / approve / reject / finish / cancel`

### 原则 2：页面权限和按钮权限分开

例如：

1. 页面进入权限：
- `asset:info:list`

2. 页面内动作权限：
- `asset:info:add`
- `asset:info:edit`

这样前端路由和页面按钮都能清晰控制。

### 原则 3：业务动作权限不要混进 CRUD

例如单据审批和盘点处理，不能只靠 `edit` 表达。

应该单独定义：

1. `asset:order:submit`
2. `asset:order:approve`
3. `asset:order:reject`
4. `asset:order:finish`
5. `asset:inventory:processDiff`

### 原则 4：角色决定“能做什么”，数据权限决定“能看哪些数据”

例如：

1. 资产管理员和部门主管都可能有 `asset:order:query`
2. 但资产管理员可以看全公司，部门主管默认只看本部门及子部门

这两层要分开理解。

## Role Design

建议第一阶段先收敛成下面 5 个角色。

### 1. 超级管理员

职责：

1. 系统级菜单维护
2. 所有资产模块查看和操作
3. 权限配置、角色配置、字典维护

权限建议：

1. 拥有全部 `asset:*`
2. 数据范围：全部

### 2. 资产管理员

职责：

1. 维护资产分类、位置、台账
2. 发起和执行业务单据
3. 发起和关闭盘点任务
4. 发起和处理维修单据
5. 查看资产流水和导出

权限建议：

1. 看板：有
2. 分类、位置、台账：增删改查导入导出
3. 单据：增删改查、提交、完成、取消、导出
4. 盘点：增删改查、启动、结束、差异处理、导出
5. 维修：查询、增删改查、提交、审批、驳回、完成、取消、导出
6. 流水：查询、导出

数据范围建议：

1. 全部或自定义部门

### 3. 部门主管

职责：

1. 查看本部门资产台账
2. 审批本部门领用、调拨、报废等单据
3. 参与盘点差异确认

权限建议：

1. 看板：有
2. 台账：查询
3. 单据：查询、审批、驳回、导出
4. 盘点：查询、差异处理
5. 维修：查询、审批、驳回、导出
6. 流水：查询

数据范围建议：

1. 本部门及以下

### 4. 资产使用人

职责：

1. 查看自己名下资产
2. 发起领用、借用、归还申请
3. 查看自己发起的单据状态

权限建议：

1. 看板：可选，只看“我的待办”
2. 台账：查询，但只看本人资产
3. 单据：查询、新增、提交
4. 维修：查询、新增、修改、删除、提交、取消

数据范围建议：

1. 本人或本部门受限视图

### 5. 财务/审计角色

职责：

1. 查看资产原值、残值、报废处置记录
2. 查看盘点差异与资产流水

权限建议：

1. 台账：查询、导出
2. 单据：查询、导出
3. 维修：查询、导出
4. 流水：查询、导出
5. 报废审批：如果业务要求，可赋予 `asset:order:approve`

数据范围建议：

1. 全部

## Data Scope Strategy

### 为什么要单独强调数据范围

因为资产台账、单据、盘点任务本质上都不是“只有有没有按钮”这么简单，而是“能看到谁的数据”。

建议第一阶段这样处理：

1. 超级管理员：全部数据
2. 资产管理员：全部数据
3. 部门主管：本部门及以下
4. 资产使用人：本人相关资产、本人发起单据
5. 财务/审计：全部数据

### 实现建议

1. 优先复用若依角色数据权限
2. 对“本人数据”这种特殊范围，在业务 SQL 层补充过滤条件
3. 不要在菜单系统里试图表达数据范围

## Suggested `sys_menu` Initialization Strategy

### 初始化顺序建议

如果后面开始写菜单初始化 SQL，建议顺序如下：

1. 新增一级目录 `资产管理`
2. 新增二级页面菜单
3. 给每个页面菜单补按钮权限
4. 绑定角色菜单关系
5. 最后联调前端 `v-auth` 和后端 `@PreAuthorize`

### 菜单 ID 规划建议

设计阶段不要求现在就定死 ID，但真正落库时应预留连续区间，避免和系统现有菜单冲突。

比如：

1. 一级目录：`3000`
2. 二级菜单：`3100 ~ 3199`
3. 按钮权限：`3200 ~ 3399`

核心目的是：

1. 资产模块菜单有自己独立号段
2. 后续新增按钮不至于到处跳号

## Frontend Mapping

### 前端路由模块建议

文件：

1. `art-design-pro/src/router/modules/asset.ts`

建议按一个独立模块组织资产菜单，而不是散落到系统模块里。

### 页面内权限使用建议

例如在资产台账页：

1. 页面显示权限依赖 `asset:info:list`
2. “新增资产”按钮依赖 `asset:info:add`
3. “编辑资产”按钮依赖 `asset:info:edit`
4. “导入资产”按钮依赖 `asset:info:import`
5. “查看流水”按钮依赖 `asset:info:viewEvent`

### 不建议的做法

1. 前端自己发明一套权限 key
2. 后端 `perms` 和前端 `authMark` 使用不同命名
3. 只控制页面，不控制接口

## Backend Mapping

后端接口权限建议全部显式标注。

例如：

1. 列表接口
- `@PreAuthorize("@ss.hasPermi('asset:info:list')")`

2. 查询详情
- `@PreAuthorize("@ss.hasPermi('asset:info:query')")`

3. 新增
- `@PreAuthorize("@ss.hasPermi('asset:info:add')")`

4. 审批
- `@PreAuthorize("@ss.hasPermi('asset:order:approve')")`

这样菜单、按钮、接口、角色授权就能完全对齐。

## Risks

### 风险 1：权限名过细过散

如果一开始就为每个单据类型都拆一套权限，比如：

1. `asset:assign:add`
2. `asset:transfer:add`
3. `asset:return:add`
4. `asset:dispose:add`

一期会很快让授权复杂度失控。

建议：

1. 先统一进 `asset:order:*`
2. 页面或服务层根据 `order_type` 再做限制

### 风险 2：角色太多

如果第一期就定义十几个角色，菜单授权会迅速变得难维护。

建议：

1. 先把 5 个核心角色跑顺
2. 后面再按企业实际组织细分

### 风险 3：普通使用人看见了不该看的台账

这通常不是按钮权限问题，而是查询条件没收紧。

建议：

1. 普通使用人的台账接口必须做“本人相关资产”过滤
2. 普通使用人的单据接口必须做“本人发起/本人相关”过滤

## Recommended Approach

最稳的下一步不是立即写一大段 `sys_menu` 初始化 SQL，而是先完成下面四件事：

1. 定菜单树
2. 定权限命名
3. 定角色边界
4. 定数据范围策略

这四件事一旦稳定，后面写 `sys_menu`、`sys_role_menu` 和前后端权限联调，都会顺很多。

## Open Questions

1. 普通使用人是否需要独立菜单入口，还是和资产管理员共用页面但只看自己的数据？
2. 报废审批权限应该归部门主管，还是财务/审计角色？
3. 盘点差异处理是否应该只给资产管理员，还是部门主管也可以操作？
4. 看板首页是否按角色展示不同卡片和待办？
