## Problem Statement

上一版已经把实施方案拆到了“表结构 + 菜单设计 + 接口清单”。这一版继续往下走，但仍然停留在设计层，不直接写 SQL 文件。

目标是回答三个更落地的问题：

1. 每张表在数据库里到底承担什么责任。
2. 表和表之间如何关联，哪些地方必须约束，哪些地方故意不做强外键。
3. 后续真正写 DDL 时，哪些字段和索引是必须保留的，哪些可以延后。

## Constraints

1. 数据库以 MySQL 为前提，风格保持和若依现有 SQL 一致。
2. 第一阶段优先保证“业务能跑通”，不是追求过度规范化。
3. 业务表尽量保留若依的审计字段与逻辑删除字段。
4. 不强依赖数据库外键做业务约束，核心一致性由服务层事务保证。

## Premises

1. `asset_info` 是资产当前状态的唯一事实来源。
2. `asset_operate_order` 是业务动作的唯一入口。
3. `asset_event_log` 是审计和追溯的唯一历史记录。
4. 盘点流程有自己独立的生命周期，因此不塞进通用业务单据。
5. 第一阶段不单独拆维修表，也不引入供应商、采购订单、财务折旧表。

## Core Relationship Map

建议把主关系理解成下面这条链：

1. 分类定义资产规则
- `asset_category` -> `asset_info`

2. 位置定义资产空间归属
- `asset_location` -> `asset_info`

3. 资产主档承接当前快照
- `asset_info`

4. 单据驱动资产变化
- `asset_operate_order` -> `asset_operate_order_item` -> `asset_info`

5. 盘点记录资产和现场差异
- `asset_inventory_task` -> `asset_inventory_task_item` -> `asset_info`

6. 所有变化沉淀为事件日志
- `asset_event_log` -> `asset_info`

7. 附件挂在资产、单据或盘点任务上
- `asset_attachment` -> `biz_type + biz_id`

## Table Responsibility

### `asset_category`

只负责分类树和分类规则，不承载资产实例数据。

适合放的内容：

1. 编码、名称、父子层级
2. 是否必须序列号
3. 是否允许借用
4. 是否纳入盘点
5. 使用寿命或默认规则

不适合放的内容：

1. 当前资产数量
2. 当前资产总值
3. 当前责任人

这些都应该通过统计查询得到，不该冗余进分类表。

### `asset_location`

只负责位置树和位置管理属性。

适合放的内容：

1. 位置编码、名称、父子结构
2. 位置类型
3. 所属部门
4. 位置管理员

不适合放的内容：

1. 位置上当前有多少资产
2. 位置上的资产总价值

这些同样应该通过聚合查询得到。

### `asset_info`

这是最重要的表，表示“现在这件资产是什么状态”。

必须具备三类字段：

1. 身份字段
- `asset_id`
- `asset_code`
- `serial_no`

2. 业务描述字段
- `asset_name`
- `category_id`
- `brand`
- `model`
- `specification`

3. 当前归属字段
- `asset_status`
- `use_org_dept_id`
- `manage_dept_id`
- `current_user_id`
- `current_location_id`

设计原则：

1. 任何时刻查询一件资产，都应该先查这张表。
2. 它是快照，不是历史表。
3. 历史变化不要在这张表保留太多“上一次值”。

### `asset_operate_order`

这张表负责“这是什么业务动作、谁发起、审批到哪了”。

它的关键作用不是存资产本身，而是存业务过程：

1. 单号
2. 单据类型
3. 单据状态
4. 发起人
5. 审批人
6. 来源归属
7. 目标归属

设计原则：

1. 单据头不直接列出所有资产字段。
2. 资产明细单独走 `asset_operate_order_item`。
3. 单据状态和资产状态必须拆开。

### `asset_operate_order_item`

这张表负责“这一张单据改了哪些资产、每件资产改成什么”。

建议保留变更前后字段，而不是只保留目标字段。这样后面审计会轻松很多。

必须具备：

1. `order_id`
2. `asset_id`
3. `before_*`
4. `after_*`
5. `item_status`

### `asset_inventory_task`

这是盘点任务头。

最重要的是定义盘点边界：

1. 盘点范围
2. 计划时间
3. 负责人
4. 汇总结果

这张表不做逐资产差异记录。

### `asset_inventory_task_item`

这是盘点最有业务价值的表。

它回答的是：

1. 账面是什么
2. 现场看到什么
3. 差异是什么
4. 差异有没有处理

如果第一阶段只能把一张表设计扎实，这张表值得花时间。

### `asset_event_log`

这张表只做一件事：记住历史。

它不承担实时查询职责，所以可以更偏审计设计：

1. 允许保留 JSON 快照
2. 允许文本说明
3. 允许来源单据回溯

但要注意：

1. 不能把所有查询都依赖它
2. 不能拿它替代业务单据

### `asset_attachment`

这张表设计尽量轻，不需要过度业务化。

一期只要解决：

1. 文件挂在哪个业务对象上
2. 文件叫什么
3. 文件地址是什么
4. 谁上传的

就够了。

## Key Constraints And Why

### 为什么不建议一期开强外键

从纯数据库规范看，很多字段都能做外键，例如：

1. `asset_info.category_id -> asset_category.category_id`
2. `asset_info.current_user_id -> sys_user.user_id`
3. `asset_operate_order_item.asset_id -> asset_info.asset_id`

但在若依这类后台项目里，一期更推荐：

1. 先做索引，不强做数据库外键
2. 在服务层做完整校验
3. 在事务里完成状态变更

原因：

1. 对历史数据导入更友好
2. 对后续数据迁移更友好
3. 不容易被一堆外键级联卡住开发节奏

### 哪些唯一约束不能省

一期最不该省的唯一约束：

1. `asset_category.category_code`
2. `asset_location.location_code`
3. `asset_info.asset_code`
4. `asset_operate_order.order_no`
5. `asset_inventory_task.task_no`

`serial_no` 是否强唯一，要看业务事实：

1. 如果企业内每台设备序列号唯一且可信，可以唯一。
2. 如果历史数据里很多设备没有序列号，或者录入质量差，就先做普通索引。

## Required Indexes

### 查询驱动索引

一期必须优先考虑列表检索性能，建议保留：

1. `asset_info(asset_status)`
2. `asset_info(category_id)`
3. `asset_info(use_org_dept_id)`
4. `asset_info(current_user_id)`
5. `asset_info(current_location_id)`
6. `asset_operate_order(order_type, order_status)`
7. `asset_operate_order(apply_user_id)`
8. `asset_inventory_task(task_status)`
9. `asset_event_log(asset_id, operate_time)`

### 不建议一期就上的索引

1. 大量组合索引
2. 每个外键字段都单独建索引
3. 所有文本字段全文索引

这些等真实查询路径稳定后再补。

## Status Flow Design

### 资产状态

推荐一期只保留这些真实状态：

1. `IDLE`
2. `IN_USE`
3. `BORROWED`
4. `IN_TRANSFER`
5. `IN_INVENTORY`
6. `PENDING_DISPOSAL`
7. `DISPOSED`

不建议把 `DRAFT` 放进正式资产主档状态里。`DRAFT` 更适合作为单据状态，而不是资产状态。

### 单据状态

推荐：

1. `DRAFT`
2. `SUBMITTED`
3. `APPROVED`
4. `REJECTED`
5. `DONE`
6. `CANCELED`

一期其实不一定需要 `APPROVING`，除非审批人不是单一节点。

### 盘点状态

推荐：

1. `DRAFT`
2. `RUNNING`
3. `FINISHED`
4. `CLOSED`

## Suggested DDL Ordering

如果后面开始正式写 SQL，推荐建表顺序如下：

1. `asset_category`
2. `asset_location`
3. `asset_info`
4. `asset_operate_order`
5. `asset_operate_order_item`
6. `asset_inventory_task`
7. `asset_inventory_task_item`
8. `asset_event_log`
9. `asset_attachment`

原因：

1. 先把主数据立住
2. 再立单据
3. 再立追踪表

## Suggested Seed Data

建议第一期准备三类初始化数据：

### 1. 字典数据

1. 资产状态
2. 单据类型
3. 单据状态
4. 盘点结果
5. 位置类型

### 2. 演示分类

1. 办公设备
2. 网络设备
3. 生产设备
4. 家具资产

### 3. 演示位置

1. 总部
2. 研发部
3. 财务部
4. 仓库
5. 会议室

## Risks

### 风险 1：把快照表和历史表混在一起

典型表现：

1. 在 `asset_info` 上不断叠加“上次责任人”“上次位置”“上次状态”
2. 最终一张表又像主档又像日志

避免方式：

1. `asset_info` 只保留当前值
2. 历史全部落 `asset_event_log`

### 风险 2：把所有业务类型做成巨型通用字段表

典型表现：

1. 单据表出现大量只在一种单据类型有意义的字段
2. 空字段越来越多

避免方式：

1. 一期先容忍少量稀疏字段
2. 一旦某业务明显变重，二期单独拆表

### 风险 3：过早做复杂审批模型

一期如果直接做多节点审批、抄送、回退、加签，会很快把实现复杂度拉满。

建议：

1. 先做单审批人或双确认
2. 保证单据状态和资产状态一致性
3. 等业务跑起来再决定是否需要 BPM

## Recommended Approach

最稳的下一步不是马上写所有 SQL，而是先完成一张“真正能指导建表”的结构图：

1. 明确每张表的主键
2. 明确每张表的唯一约束
3. 明确必须索引
4. 明确状态字典
5. 用 3 条真实流程演练字段是否够用

如果这 5 件事都通过，再开始写 DDL，后面返工会小很多。

## Open Questions

1. `asset_info.serial_no` 是否要做唯一约束？
2. 盘点是否允许出现“现场新增资产”，也就是 `SURPLUS` 但系统无主档？
3. 报废是否需要在一期保留处置金额字段？
4. 借用是否一定需要预计归还日期？
