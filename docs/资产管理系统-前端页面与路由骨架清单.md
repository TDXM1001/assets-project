## 目标

这份文档把资产管理系统继续推进到前端实施层，重点回答四件事：

1. 资产模块在当前 `art-design-pro` 里应该落到哪些目录
2. 页面、路由、API 文件应该怎么拆
3. 哪些页面应该优先做列表页骨架，哪些要从一开始就按流程页来设计
4. 在“不使用代码生成器”的前提下，前端怎样和后端手写接口保持稳定对齐

## 项目基线

当前前端目录可以确认：

1. 路由模块在 `art-design-pro/src/router/modules`
2. 页面目录在 `art-design-pro/src/views`
3. API 按业务域拆在 `art-design-pro/src/api/*`
4. 权限指令使用 `v-auth`，实现位于 `art-design-pro/src/directives/business/permi.ts`

现状里已经存在：

1. `art-design-pro/src/views/asset`
2. `art-design-pro/src/views/asset/ledger`

但目前 `asset/ledger` 还是空壳，没有形成稳定页面结构。

所以这一轮的建议是：

1. 不沿着 `ledger` 继续发散
2. 统一收敛到 `src/views/asset/*`
3. 让路由、页面、API、权限命名都和后端 `/asset/**` 完整对齐

## 当前建议

### 建议 1：单独新增 `router/modules/asset.ts`

当前 `src/router/modules/index.ts` 只挂了：

1. `dashboard`
2. `system`
3. `result`
4. `exception`

资产模块不应该继续塞进 `system.ts`，更稳的做法是新增：

1. `art-design-pro/src/router/modules/asset.ts`

然后在：

1. `art-design-pro/src/router/modules/index.ts`

里统一导出。

### 建议 2：页面全部收敛在 `src/views/asset`

建议结构如下：

1. `src/views/asset/dashboard`
2. `src/views/asset/category`
3. `src/views/asset/location`
4. `src/views/asset/info`
5. `src/views/asset/order`
6. `src/views/asset/inventory`
7. `src/views/asset/event`

不建议再保留一个语义模糊的：

1. `src/views/asset/ledger`

如果你后面确实只想保留“资产台账”这个名字，那么也建议直接用：

1. `src/views/asset/info`

这样和后端 `/asset/info/*`、菜单 `asset:info:*`、文档里的“资产台账”都一致。

### 建议 3：API 按领域拆文件，不要堆成一个大 `asset.ts`

建议新增：

1. `src/api/asset/category.ts`
2. `src/api/asset/location.ts`
3. `src/api/asset/info.ts`
4. `src/api/asset/order.ts`
5. `src/api/asset/inventory.ts`
6. `src/api/asset/event.ts`
7. `src/api/asset/dashboard.ts`
8. `src/api/asset/attachment.ts`

### 建议 4：页面模块按“列表页骨架”和“流程页骨架”两类写

分类、位置、台账这类页面，适合复用系统管理页面的“搜索栏 + 表格 + 弹窗”模式。

单据、盘点这类页面，从第一版开始就要接受它不是普通 CRUD 页面，而是：

1. 列表页
2. 详情页/抽屉
3. 动作弹窗
4. 状态条
5. 流程按钮

## 权限接线建议

当前项目已使用：

1. `v-auth="'system:user:add'"`

资产模块建议统一切到完整后端权限标识，不要只写简短动作名。

例如：

1. `v-auth="['asset:info:add']"`
2. `v-auth="['asset:order:approve']"`
3. `v-auth="['asset:inventory:processDiff']"`

不建议前端自己发明：

1. `assetInfoAdd`
2. `approveOrder`
3. `canProcessDiff`

这种自定义 key。

## 当前不建议做的事

### 1. 不建议现在继续沿用 `asset/ledger`

因为它现在没有形成真正页面骨架，继续沿着这个名字扩展，后面很容易和“资产台账 / 资产流水 / 业务单据”混淆。

### 2. 不建议一开始把单据和盘点做成很多独立详情页路由

一期更稳的方式还是：

1. 一个列表页
2. 若干弹窗
3. 一个详情抽屉

等流程稳定后，再考虑拆独立详情页。

### 3. 不建议页面内写死大量状态文案

资产状态、单据状态、盘点状态最好都走字典或统一映射，避免后面前后端改一个状态要改很多地方。

## Phase 1 设计评审收敛结论

这一节是按 `plan-design-review` 视角补齐的实现前规范，目标不是继续讨论目录结构，而是把 `Phase 1` 真正需要落地的用户界面决定说清楚。

本轮评审结论：

1. `Phase 1` 只聚焦三个页面：`asset/category`、`asset/location`、`asset/info`
2. `asset/category` 与 `asset/location` 属于树形主数据页，但两者不能只是字段不同的复制页
3. `asset/info` 是一期最重要的工作台页面，必须从一开始就按“筛选区 + 工作区 + 详情抽屉”设计，而不是普通 CRUD 列表页
4. 一期不追求品牌化视觉创新，优先与现有 `art-design-pro` 管理端保持一致、克制、清晰

### 初始评分

当前这份前端计划在进入本轮评审前的设计完整度约为 `6/10`。

扣分点主要在：

1. 没有明确三页的信息层级
2. 没有定义加载/空态/错误态
3. 没有明确响应式退化策略
4. 没有说明如何避免把三个页面做成三张几乎一样的“后台模板页”

这份计划在 `Phase 1` 想达到 `10/10`，至少要满足：

1. 每页都有明确的首要任务和主次信息层级
2. 每页都有状态覆盖，不出现“暂无数据”这种空壳默认文案
3. 桌面端、窄屏、键盘操作和屏幕阅读器行为都有明确规范
4. 页面复用现有系统组件，但不会陷入机械复制

## Pass 1：Information Architecture

### 评分

调整前：`5/10`

问题在于当前计划只说明“页面有哪些模块”，但没有定义用户进入页面后第一眼看到什么、什么是主操作、什么信息可以后置。

调整后目标：`9/10`

### Phase 1 总体层级

`Phase 1` 三个页面的主次关系不一样，必须区别对待：

1. `category`：以“结构维护”为主
2. `location`：以“结构维护 + 责任归属查看”为主
3. `info`：以“资产查询、筛选、定位、变更入口”为主

### 页面 1：分类管理 `asset/category`

页面目标：

1. 让资产管理员快速理解分类树
2. 快速新增、修改、停用分类
3. 快速判断某分类是否允许借用、是否要求序列号、是否纳入盘点

桌面端信息层级：

1. 第一层：页面标题 + 页面级主操作 `新增一级分类`
2. 第二层：紧凑搜索栏，只保留 `分类名称`、`状态`
3. 第三层：树表主体
4. 第四层：行内操作 `新增下级 / 编辑 / 删除`

ASCII 结构：

```text
+---------------------------------------------------------------+
| 分类管理                                     [新增一级分类]   |
+---------------------------------------------------------------+
| 搜索：分类名称 [____] 状态 [____] [查询] [重置]              |
+---------------------------------------------------------------+
| 分类树表                                                       |
| ├─ 办公设备      借用:是  盘点:是  序列号:是   [新增] [编辑] |
| │  ├─ 笔记本电脑                                           |
| │  └─ 台式电脑                                             |
| └─ 家具类                                                  |
+---------------------------------------------------------------+
```

设计决定：

1. 不做左树右详情，因为分类本身就是树，左树右树会造成双重层级
2. 不做独立详情页，一期所有编辑都在弹窗完成
3. 树表列里直接显示行为属性，不把关键属性藏进弹窗

### 页面 2：存放位置 `asset/location`

页面目标：

1. 让管理员快速维护位置树
2. 快速看出位置与部门、管理员的关系
3. 为后续资产台账筛选提供稳定的位置主数据

桌面端信息层级：

1. 第一层：页面标题 + 页面级主操作 `新增顶级位置`
2. 第二层：紧凑搜索栏，只保留 `位置名称`、`位置类型`、`状态`
3. 第三层：树表主体
4. 第四层：辅助字段 `所属部门 / 管理员 / 状态`

ASCII 结构：

```text
+-------------------------------------------------------------------+
| 存放位置                                         [新增顶级位置]   |
+-------------------------------------------------------------------+
| 搜索：位置名称 [____] 类型 [____] 状态 [____] [查询] [重置]      |
+-------------------------------------------------------------------+
| 位置树表                                                           |
| ├─ 总部大楼    类型:楼栋  所属:行政部  管理员:张三   [新增][编辑] |
| │  ├─ 3F      类型:楼层                                          |
| │  │  └─ 301  类型:房间                                          |
| └─ 一仓       类型:仓库                                          |
+-------------------------------------------------------------------+
```

设计决定：

1. 相比分类页，多显示“所属部门 / 管理员”两列，强调它是组织资源，不只是树结构
2. 不新增“位置详情页”，避免一期过早分裂流程
3. 行内操作顺序保持与分类页一致，降低学习成本

### 页面 3：资产台账 `asset/info`

页面目标：

1. 让用户最快找到一批资产
2. 让用户快速判断资产当前归属、状态、位置
3. 让新增、编辑、导入、看流水四类动作都能在同一工作台完成

桌面端信息层级：

1. 第一层：页面标题 + 主操作区 `新增资产 / 导入 / 导出`
2. 第二层：左侧筛选侧栏
3. 第三层：顶部搜索栏
4. 第四层：表格主体
5. 第五层：详情抽屉 / 流水抽屉

推荐布局：

1. 左侧不是同时放“分类树 + 位置树”
2. 左侧使用一个可切换的筛选侧栏：`按分类` / `按位置`
3. 右侧维持标准列表工作台

ASCII 结构：

```text
+--------------------------------------------------------------------------------+
| 资产台账                       [新增资产] [导入] [导出]                         |
+----------------------+---------------------------------------------------------+
| 左侧筛选侧栏         | 搜索栏                                                  |
| [按分类] [按位置]    | 编码[__] 名称[__] 状态[__] 责任人[__] [查询][重置]     |
| 搜索树节点 [____]    +---------------------------------------------------------+
| 办公设备             | 表格                                                    |
| 家具类               | 编码  名称  状态  分类  部门  责任人  位置  [编辑][流水]|
| ...                  |                                                         |
+----------------------+---------------------------------------------------------+
```

设计决定：

1. `asset/info` 复用 `system/user` 的左树右表工作台模式，但树侧栏改成“分类 / 位置”切换，不直接照搬部门树
2. 搜索栏不放太多字段，一期最多 5 个，避免把核心视线压扁
3. 资产详情不做新路由，先用抽屉承接
4. 查看流水是只读抽屉，不和编辑弹窗混用

## Pass 2：Interaction State Coverage

### 评分

调整前：`3/10`

问题在于原文几乎没有状态设计，工程师很容易默认输出：

1. 空列表直接显示空表格
2. 错误只打控制台
3. 成功只弹一句 toast

调整后目标：`9/10`

### Phase 1 状态覆盖表

| 功能 | Loading | Empty | Error | Success | Partial |
|---|---|---|---|---|---|
| 分类管理页 | 树表区域显示骨架行，新增按钮保留可见 | 显示“还没有分类”空态，主按钮为“新增一级分类” | 卡片内错误提示 + “重新加载”按钮 | 新增/编辑成功后高亮对应行并保留树展开状态 | 搜索无结果时显示“没有匹配分类”，不是“系统空” |
| 存放位置页 | 树表区域显示骨架行 | 显示“还没有位置结构”，主按钮为“新增顶级位置” | 卡片内错误提示 + “重新加载”按钮 | 新增/编辑成功后滚动到对应节点 | 搜索无结果时提示可尝试按管理员/所属部门调整条件 |
| 资产台账页 | 左侧树和右侧表格分别 loading，避免整页锁死 | 空库时显示说明型空态，主按钮为“新增资产”，次按钮为“导入资产” | 表格区错误状态 + 重新加载；树侧栏失败不应拖垮右侧搜索栏 | 新增/编辑成功后保留当前筛选与分页 | 树侧栏有数据但表格为空时，文案改为“当前筛选下暂无资产” |
| 资产导入弹窗 | 上传中显示进度和文件名 | 未选择文件时提示支持格式与模板下载 | 导入失败显示失败条数与错误原因入口 | 导入成功展示成功条数并给出“查看最新导入数据” | 部分成功时显示成功/失败拆分结果 |
| 资产流水抽屉 | 抽屉内显示时间线骨架 | 显示“这项资产还没有流水记录” | 抽屉内错误态，不关闭抽屉 | 成功展示最近事件时间线 | 只拿到部分事件时保留“继续加载”入口 |

### 空态文案要求

禁止直接使用：

1. `暂无数据`
2. `No Data`
3. `列表为空`

一期推荐空态语气：

1. 说明当前为什么空
2. 给一个主动作
3. 必要时给一个低成本教育提示

示例：

1. 分类页：`还没有资产分类，先建立分类树，后面建账才不会混乱。`
2. 位置页：`还没有存放位置，先把楼栋、楼层、房间或仓库结构补齐。`
3. 资产台账页：`还没有资产台账。你可以先新增一条资产，或者批量导入现有台账。`

## Pass 3：User Journey & Emotional Arc

### 评分

调整前：`4/10`

原文关注了“页面怎么拆”，但没有关注用户第一次进入和持续使用时的感受。

调整后目标：`8/10`

### Phase 1 关键旅程

| 步骤 | 用户在做什么 | 用户感受 | 页面需要支持什么 |
|---|---|---|---|
| 1 | 第一次进入资产模块 | 需要快速判断从哪开始 | 资产台账页空态必须明确指向“新增资产 / 导入资产”；分类、位置页空态要告诉用户这是前置准备 |
| 2 | 先建分类和位置 | 希望结构清楚，不想迷路 | 树表必须保持展开/收起可控，新增下级动作必须足够直观 |
| 3 | 回到资产台账查资产 | 希望快速定位，不想被太多筛选器干扰 | `asset/info` 首屏以筛选和列表为主，不堆统计卡片 |
| 4 | 编辑某条资产 | 需要确认改动不会迷失当前上下文 | 编辑完成后返回原筛选、原分页、原滚动位置 |

### 情绪设计要求

1. 分类页和位置页应该给人“结构清楚、可维护”的感觉，而不是“功能很多”
2. 资产台账页应该给人“可查、可信、可追溯”的感觉，而不是“花哨”
3. 所有成功反馈都以“完成操作 + 保留上下文”为核心，不强制整页刷新回起点

## Pass 6：Responsive & Accessibility

### 评分

调整前：`2/10`

原文几乎没有响应式和无障碍描述，这类后台页面最容易被默认成“PC 能看就行”。

调整后目标：`8/10`

### 响应式规范

#### Desktop `>= 1280`

1. 分类页、位置页：单列树表布局
2. 资产台账页：左侧固定宽度筛选侧栏 + 右侧表格工作区
3. 表格按主要字段完整展示，不主动折叠主信息

#### Laptop / Tablet `768 - 1279`

1. 分类页、位置页：保持树表，但默认不全展开
2. 资产台账页：左侧筛选侧栏默认收起，使用按钮展开为覆盖层或抽屉
3. 表格次要列可进入列设置，不强行在首屏全展示

#### Mobile `< 768`

1. 分类页、位置页：不显示常驻树侧栏，改为顶部筛选 + 折叠树弹层
2. 资产台账页：左侧树筛选改为顶部按钮触发的抽屉
3. 弹窗优先切为全屏抽屉或全屏对话框
4. 表格允许只保留关键字段：编码、名称、状态、责任人

### 无障碍要求

1. 所有可点击按钮最小触达尺寸 `44px`
2. 树筛选支持键盘上下切换和回车选中
3. 搜索栏输入项有明确 `label` 或等价语义
4. 抽屉打开后焦点进入抽屉，关闭后焦点回到触发按钮
5. 状态不能只靠颜色区分，必须同时有文字
6. 错误状态需要在视觉提示之外提供明确文本
7. 对比度沿用现有主题变量，不新增低对比浅灰占位文本作为主要信息

## Pass 7：Phase 1 关键设计决定

本轮最后一个关键选择已经正式定稿：

1. `asset/info` 页面左侧采用“单侧栏切换分类 / 位置”的方案

### 已确认方案

固定规则如下：

1. 左侧始终只存在一个筛选侧栏
2. 侧栏顶部提供两个切换入口：`按分类`、`按位置`
3. 同一时刻只展示一棵树，避免双树并列
4. 当前选中的筛选模式需要在页面会话内保留
5. 窄屏下该侧栏退化为顶部按钮触发的抽屉

原因：

1. 更贴近 `system/user` 的现有工作台模式
2. 比双树并列更稳，更容易控制桌面端宽度
3. 后续做抽屉化和移动端退化时改动最小

### 对工程实现的直接约束

1. `asset/info/index.vue` 必须按“单侧栏 + 工作区”布局起骨架
2. 不允许在一期实现里把分类树和位置树同时固定渲染到左侧
3. 搜索栏条件与左侧树筛选是互补关系，不重复堆叠同类入口
4. 左侧切换不会触发整页重建，只更新当前筛选树和列表查询参数

## Phase 1 实施准则

这三页实现时按下面的优先级推进：

1. 先把信息层级做对
2. 再把状态覆盖做全
3. 最后再做视觉收口

禁止的做法：

1. 先把三个页面都复制出来再慢慢修
2. 首屏先塞统计卡片和图表
3. 只做 happy path，不定义空态和错误态
4. 资产台账页把分类树和位置树同时钉在页面左侧

## NOT in Scope

本轮明确不纳入 `Phase 1` 的前端设计项：

1. 资产看板的视觉强化
原因：一期主目标是主数据可用，不是看板表现力。

2. 单据与盘点独立详情页路由
原因：一期先用列表 + 抽屉跑通流程，避免导航复杂化。

3. 复杂批量操作、列设置、个性化视图
原因：这些属于效率增强，不是一期闭环必需项。

4. 移动端深度优化
原因：一期只要求可退化可使用，不要求移动端成为主工作场景。
