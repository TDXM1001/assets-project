# 资产管理系统下一阶段功能建议

> 这份文档是基于前面几轮讨论的收敛稿，回答一个更直接的问题：
> 当前资产管理系统核心闭环已经搭起来了，下一步除了修前端页面，还应该优先做哪些功能？

## 1. 先说结论

当前系统已经完成了分类、位置、台账、单据、盘点、维修、流水、看板和附件的主骨架。  
下一阶段不建议继续把精力放在零碎 UI 修补上，而是优先做这三类能力：

1. 页面化复杂录入流程
2. 分类驱动的字段模板
3. 多资产和批量业务能力

这三类能力会直接决定系统后面能不能继续扩展到不动产、低值易耗品、扫码、移动盘点等场景。

---

## 2. BMAD 对齐说明

这份文档是 **BMAD 的补充设计说明**，用于在 PRD、架构和 story 之外补齐产品判断。

它的作用是：

1. 帮助确定下一轮要做什么
2. 帮助确定哪些功能应该进入新 story
3. 帮助区分“现在就做”和“以后再做”

它不替代：

1. PRD
2. 架构
3. 用户故事

---

## 3. 下一阶段优先级建议

### P0：必须优先做

#### 3.1 资产新增页面化

建议把“新增资产”从弹窗升级为独立页面或向导页。

原因：

1. 字段已经很多
2. 后面还要接分类模板
3. 还会叠加附件、草稿、批量补录
4. 弹窗已经接近复杂度上限

页面应支持：

1. 基础信息
2. 归属信息
3. 价值信息
4. 分类扩展字段
5. 附件

#### 3.2 单据新增页面化

建议把入库、领用、归还、调拨、报废等单据，统一成页面化新增流程。

原因：

1. 单据有头信息和明细
2. 单据需要保存草稿、提交、审批、完成
3. 复杂单据不适合长期停留在弹窗里

页面应支持：

1. 单据类型选择
2. 单据头信息
3. 业务对象信息
4. 资产明细
5. 附件与备注

#### 3.3 分类字段模板

建议把“分类决定哪些字段出现、是否必填、用什么组件展示”做成模板化能力。

这是后面所有扩展能力的底座。

模板建议支持：

1. 必填
2. 只读
3. 默认值
4. 字典来源
5. 组件类型
6. 排序
7. 分组
8. 版本号

### P1：应该尽快做

#### 3.4 多资产维修

建议支持一张维修单包含多个资产，但每个资产必须有独立明细。

适合场景：

1. 批量送修
2. 同一供应商维修
3. 同一批次返修

不适合场景：

1. 每个资产故障和结果差异很大
2. 费用和完成时间差异很大

#### 3.5 盘点增强

建议把盘点任务做得更完整一些，包括：

1. 盘点任务创建
2. 盘点明细执行
3. 差异处理
4. 盘点结果回写

#### 3.6 操作栏统一

建议统一顶部按钮区和行内按钮区的样式与语义。

规则建议：

1. 顶部主操作保留按钮
2. 行内操作统一文字按钮
3. 危险操作统一红色样式
4. 次级操作统一收进“更多”

### P2：可以作为扩展项

#### 3.7 资产类型扩展

建议后续支持更多资产类型，例如：

1. 固定资产
2. 不动产
3. 低值易耗品

这类扩展不建议直接硬塞进当前固定资产逻辑里，而是放在“类型 + 模板 + 流程”的框架下扩。

#### 3.8 扫码 / 二维码 / 移动端辅助

适合盘点、查找、快速确认资产归属。

建议在主流程稳定后再做。

#### 3.9 预警与统计增强

建议补齐：

1. 即将到期保修
2. 待审批
3. 维修积压
4. 盘点异常
5. 异常资产趋势

---

## 4. 不建议现在优先做的事

### 4.1 复杂审批引擎

当前系统还不需要把审批做成重型流程平台。  
先把业务单据跑通，比先做复杂流程引擎更重要。

### 4.2 完全自由的低代码表单

如果直接做成无限可配的低代码系统，后续会失控。

更稳的方式是：

1. 固定核心字段
2. 分类模板受控扩展
3. 重要字段保持表列

### 4.3 继续堆状态枚举

资产状态不要无限增加。  
状态应该保持少量稳定，业务差异交给单据、流水和模板表达。

---

## 5. 推荐的下一轮故事顺序

如果要按 BMAD 的 story 方式继续，我建议优先顺序是：

1. 资产新增页面化
2. 单据新增页面化
3. 分类字段模板
4. 多资产维修
5. 盘点增强
6. 操作栏统一

如果你只准备做一轮，我建议直接做前 3 个。

原因很简单：

1. 它们会直接抬高系统能力上限
2. 它们会为后续扩展打底
3. 它们对现有模块的返工成本最低

---

## 6. 一句话总结

**下一阶段最值得做的，不是继续修碎 UI，而是把“资产新增、单据新增、字段模板”三件事做成系统能力。**
