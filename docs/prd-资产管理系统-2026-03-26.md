## 1. 背景

当前仓库已经具备 `RuoYi-Vue` 后端底座和 `art-design-pro` 前端底座，但资产业务仍处在空白状态。当前项目的目标不是做一个普通 CRUD 台账，而是建立一套可追踪、可审批、可盘点、可追责的企业内部资产管理系统。

系统需要覆盖资产从入库建账、领用、归还、调拨、盘点到报废处置的核心闭环，并且让“资产归谁、在哪、什么状态、何时变更、谁改的”都能被系统明确记录。

## 2. 产品目标

| 目标 ID | 目标 | 成功标准 |
|---|---|---|
| G1 | 统一资产台账 | 任一资产都能查到唯一编码、分类、责任部门、责任人、当前位置和当前状态 |
| G2 | 跑通核心生命周期 | 入库、领用、归还、调拨、盘点、报废都能形成单据和流水 |
| G3 | 建立审计追踪 | 所有关键状态变化都写入事件流水，能追溯前后值、操作者和来源单据 |
| G4 | 降低人工维护成本 | 资产管理员不再依赖 Excel 拼接台账和变更记录 |
| G5 | 支撑管理视角 | 管理层能看到待办、异常、盘点差异和资产概况 |

## 3. 范围与优先级

| 优先级 | 范围 |
|---|---|
| Must | 资产分类、资产主档、入库建账、领用、归还、调拨、盘点任务、报废处置、资产看板、资产流水、菜单权限、数据范围 |
| Should | 导出、基础导入、详情抽屉、操作记录、基础附件位、列表筛选优化 |
| Could | 借用、维修、二维码标签、批量导入导出、更加细粒度的数据权限 |
| Won't | 采购系统、财务折旧、移动盘点、扫码枪、对外销售/租赁能力 |

## 4. 角色与典型场景

| 角色 | 核心任务 | 典型场景 |
|---|---|---|
| 超级管理员 | 配置菜单、角色、字典和全局权限 | 系统初始化和全局维护 |
| 资产管理员 | 维护台账、发起和执行单据、处理盘点差异 | 新资产入库、调拨、报废、盘点 |
| 部门主管 | 审批部门内关键单据、确认调拨和盘点差异 | 领用审批、调拨确认、报废审批 |
| 资产使用人 | 提交领用、归还、查看本人相关资产 | 发起领用申请、报修申请、归还登记 |
| 财务/审计 | 查看价值、处置、盘点差异和流水 | 审计抽查、导出流水、核对报废记录 |

## 5. 功能需求

### Epic A. 资产基础数据

| ID | 优先级 | 需求 | 验收标准 |
|---|---|---|---|
| FR-01 | Must | 支持资产分类管理，包含大类、小类、折旧年限、是否需要序列号、是否允许借用、是否需要盘点等字段 | 可新增、编辑、停用分类；分类树能正确回显 |
| FR-02 | Must | 支持存放位置管理，位置可用于部门、地点、区域等层级化组织 | 可按树结构查询位置，台账和单据能引用位置 |
| FR-03 | Must | 支持资产主档管理，资产必须具备唯一编码 | 台账可查询、新增、编辑、删除，且可按编码/分类/责任人/位置筛选 |

### Epic B. 资产入库与领用

| ID | 优先级 | 需求 | 验收标准 |
|---|---|---|---|
| FR-04 | Must | 支持入库建账，资产可由手工录入或批量导入进入系统 | 入库完成后生成资产主档，并写入事件流水 |
| FR-05 | Must | 支持领用流程，领用后资产责任人、部门、位置和状态同步更新 | 领用单可完成提交、审批、发放，台账状态同步为在用 |
| FR-06 | Must | 支持归还流程，归还后可回到闲置、在用或维修中等状态 | 归还单完成后，台账与事件流水同步更新 |

### Epic C. 调拨与处置

| ID | 优先级 | 需求 | 验收标准 |
|---|---|---|---|
| FR-07 | Must | 支持跨部门调拨，调拨可改变责任部门、责任人和位置 | 调拨完成后台账更新，且保留前后快照 |
| FR-08 | Must | 支持报废处置流程，包含申请、审批、执行和结果留痕 | 报废完成后资产进入已报废状态，禁止继续流转 |

### Epic D. 盘点与差异处理

| ID | 优先级 | 需求 | 验收标准 |
|---|---|---|---|
| FR-09 | Must | 支持按部门、地点或分类发起盘点任务 | 可批量生成盘点明细，盘点任务可开始、结束、导出 |
| FR-10 | Must | 支持盘点差异处理，至少能处理位置不符、责任人不符、账实不符等情况 | 差异处理后可回写台账，并保留盘点明细和处理结果 |

### Epic E. 运营视图与追踪

| ID | 优先级 | 需求 | 验收标准 |
|---|---|---|---|
| FR-11 | Must | 提供资产运营看板，展示资产总数、在用数、闲置数、维修中、待盘点、待审批和异常数 | 看板数据与台账口径一致，不是空壳页面 |
| FR-12 | Must | 提供资产事件流水查询，支持追溯关键动作的前后值、操作者和来源单据 | 任一关键变更都能在流水中查到 |
| FR-13 | Must | 提供菜单权限和数据范围控制，复用若依 `sys_menu`、`sys_role`、`sys_role_menu` 模型 | 不同角色看到的页面、按钮和数据范围符合设计 |
| FR-14 | Should | 提供列表导出和基础筛选能力 | 主要列表页支持导出，筛选条件能正确影响查询结果 |

## 6. 非功能需求

| ID | 类别 | 需求 | 验收标准 |
|---|---|---|---|
| NFR-01 | 性能 | 常用列表页和查询接口应保持稳定响应 | 正常数据量下，常用查询接口应在可接受时间内返回，不出现明显卡顿 |
| NFR-02 | 可追溯 | 所有状态变化必须写入事件流水 | 任何入库、领用、归还、调拨、盘点处理、报废都能追溯 |
| NFR-03 | 一致性 | 主档、单据、流水之间必须保持一致 | 单据完成后资产主档和流水同步更新，不能只改一处 |
| NFR-04 | 安全 | 接口权限、按钮权限、数据范围必须同时生效 | 没有权限的用户不能看到按钮，也不能通过接口越权 |
| NFR-05 | 可用性 | 页面需要具备空状态、异常态和加载态 | 用户在数据缺失或请求失败时能得到明确反馈 |
| NFR-06 | 可维护性 | 权限命名、菜单命名、接口命名保持统一 | 前后端权限标识一致，后续扩展不需要重新命名 |

## 7. 设计决策与默认假设

1. 第一阶段只做固定资产，不把低值易耗品、采购、折旧、供应商和合同一起纳入范围。
2. 业务闭环以“主档 + 单据 + 流水”为核心，不直接把业务动作写成主档编辑。
3. 审批流先做部门规则化配置，不引入 BPM 引擎。
4. 数据范围优先复用若依的角色数据权限模型，特殊的“本人范围”在业务 SQL 层补充过滤。
5. 资产编码默认由系统生成，历史编码可在导入或初始化时兼容。

## 8. 需要后续确认的问题

1. 普通使用人是否需要独立菜单入口，还是和资产管理员共用页面但只看自己的数据。
2. 报废审批是否由部门主管负责，还是由财务/审计角色参与最终确认。
3. 盘点差异处理是否只允许资产管理员执行，还是部门主管也可以操作。

## 9. 验收口径

1. 资产可以建账、查询、编辑和导出。
2. 领用、归还、调拨、报废可以形成完整单据和流水。
3. 盘点任务可以创建、执行、处理差异并回写台账。
4. 看板可以真实反映待办、异常和统计信息。
5. 不同角色进入系统后，页面、按钮和数据范围基本正确。

## 10. 交付建议

1. 先完成资产分类、位置和台账基础能力，再推进单据和盘点。
2. 先跑通固定资产 MVP，再补借用、维修、二维码和更细粒度权限。
3. PRD 定稿后进入架构设计，随后拆分到可执行的实施阶段。
