# 资产管理系统产品演进与交互规范建议

> 这份文档整理的是前面几轮讨论的产品结论，重点回答四件事：
> 1. 资产生命周期是不是只能停留在当前状态集合
> 2. 维修是否可以一单多资产
> 3. 新增台账、单据、盘点、维修是否应该独立页面
> 4. 前端操作栏是否要统一规范，以及系统是否还能继续扩展

## 1. 结论先行

1. 资产生命周期不是只能停留在当前这几个状态。
2. 维修可以支持一单多资产，但必须按明细逐个落地。
3. 新增台账、单据、盘点、维修这类复杂操作，建议从弹窗升级为独立页面或向导页。
4. 前端操作栏应该统一规范，但不是把所有按钮都改成同一种样式。
5. 系统可以继续扩展，但扩展方式应该是“类型 + 模板 + 流程”，不是无限给核心表加字段。
6. 这份文档适合作为 BMAD 的补充设计说明，不替代 PRD、架构和 story。

---

## 2. 资产生命周期应该怎么理解

当前系统里的资产状态，已经足够支撑内部固定资产管理的 MVP，但它不是终点。

更合理的拆法是把资产管理拆成四层：

1. 资产类型
2. 资产状态
3. 业务动作
4. 事件流水

### 2.1 建议的状态边界

资产状态只表示“资产当前真实处于什么状态”，不要混入审批状态。

建议保持在少量稳定状态内，例如：

- 闲置
- 在用
- 借出
- 维修中
- 盘点中
- 待报废审批
- 已报废

审批过程、流转过程、驳回过程，应该放到单据状态里表达。

### 2.2 建议的类型边界

如果后续要纳入不动产，不建议直接塞进当前固定资产生命周期。

更稳的方式是增加资产类型：

- 固定资产
- 不动产
- 低值易耗品
- 其他资产

不同类型下面再挂不同分类树、字段模板和流程配置。

### 2.3 结论

不是“生命周期状态越多越好”，而是：

- 核心状态要少
- 流程动作要完整
- 追溯流水要全

这样系统才不会越做越乱。

## 3. 维修是否可以一单多资产

可以，而且建议支持。

现实里维修经常是批量发生的，比如一批笔记本、一批显示器、同一批网络设备一起送修。

但要注意边界：

1. 一个维修单可以包含多个资产
2. 每个资产必须有独立明细
3. 明细必须记录各自的故障、费用、结果和回写状态

### 3.1 推荐的数据结构

- 单据头：维修原因、供应商、申请人、审批信息、维修方式
- 单据明细：资产 ID、故障描述、费用、结果、是否修复、是否建议报废

### 3.2 什么情况下要拆单

如果同一批资产：

- 故障类型差异很大
- 维修结果差异很大
- 费用差异很大
- 完成时间差异很大

那就不应该强行放在一张单里，应该拆成多个维修单。

### 3.3 结论

维修支持批量，但要“批次化 + 明细化”，不要做成一笔模糊的大单。

## 4. 新增台账和单据是否要独立页面

我的建议是：**要。**

而且不是只有台账新增要独立页面，以下这几类也建议逐步页面化：

- 新增资产
- 新增单据
- 新增盘点任务
- 新增维修单

### 4.1 为什么不能一直用弹窗

因为这些表单都已经进入复杂区间，字段多、分区多、联动多：

- 资产台账有分类、状态、部门、责任人、位置、日期、金额、附件
- 单据有头信息、明细、审批、完成、取消
- 盘点有范围、明细、差异处理
- 维修有批次、明细、结果、回写

如果继续把它们塞在弹窗里，后面会越来越挤，越来越难维护。

### 4.2 推荐的页面形态

| 场景 | 推荐形态 | 原因 |
|---|---|---|
| 分类、位置维护 | 弹窗 | 结构简单，树形维护轻量 |
| 台账新增/编辑 | 独立页面或向导页 | 字段多，后续还会接动态模板 |
| 单据新增/编辑 | 独立页面 | 有明细、有流转、有上下文 |
| 盘点创建/处理 | 独立页面 | 任务型流程，信息密度高 |
| 维修创建/处理 | 独立页面 | 常涉及多资产和结果分流 |
| 审批确认/驳回 | 弹窗或抽屉 | 动作短，保留快捷性即可 |

### 4.3 建议保留的弹窗

弹窗不要完全取消，建议保留给这些轻量场景：

- 快速编辑
- 审批确认
- 驳回原因填写
- 简单补录

也就是说，弹窗保留为“快捷入口”，主流程页面化。

## 5. 前端操作栏怎么统一

建议统一，但统一的是“规则”，不是“全部改成一样的按钮”。

### 5.1 推荐的统一原则

1. 顶部主操作区保留按钮层级。
2. 列表行内操作统一为文字按钮或文字 + 更多菜单。
3. 危险操作统一使用红色样式。
4. 主操作保留 `primary`，不要全部降成文字按钮。
5. 每个列表页最多保留少量显式主按钮，其余动作收进“更多”。

### 5.2 推荐的操作栏结构

#### 顶部操作栏

- 新增 / 新建
- 导入
- 导出
- 批量处理

建议保留主按钮样式，并且在长列表页做成固定或半固定区域，减少滚动后找按钮的成本。

#### 行内操作栏

- 查看
- 编辑
- 提交
- 审批
- 完成
- 删除
- 更多

建议尽量统一成文字按钮，减少视觉噪音。

### 5.3 为什么不要“全部文字按钮化”

因为这样会丢失层级：

- 新增是主操作
- 导出是次要操作
- 删除是危险操作
- 审批、完成是业务动作

如果都做成一样，用户会分不清主次。

### 5.4 结论

最佳做法是：

- 顶部主操作：按钮
- 行内操作：文字按钮
- 危险操作：红色文字
- 次级动作：更多菜单

## 6. 系统还能不能继续扩展

可以，而且应该扩，但要控制扩展方式。

### 6.1 推荐的扩展路径

#### 路径 A：扩类型

先引入新的资产类型，例如：

- 固定资产
- 不动产
- 低值易耗品

#### 路径 B：扩模板

再给不同类型、不同分类配置字段模板：

- 是否必填
- 组件类型
- 字典来源
- 默认值
- 校验规则

#### 路径 C：扩流程

最后再根据类型补流程：

- 不动产登记
- 房屋维修
- 资产折旧
- 资产处置

### 6.2 不建议的扩展方式

不建议直接在 `asset_info` 上无限加列，也不建议把所有分类差异都硬编码到前端页面里。

那样会带来三个问题：

1. 表越来越宽
2. 页面越来越难维护
3. 后续扩展会越来越依赖人工改代码

### 6.3 对字段扩展的建议

适合做成动态字段的：

- 只在部分分类出现的字段
- 描述性字段
- 不经常参与筛选和统计的字段

不适合做成动态字段的：

- 高频查询字段
- 业务判断字段
- 审批关键字段
- 报表统计字段

### 6.4 结论

系统可以扩展，但扩展的正确方式是：

**核心字段固定，分类规则下沉，动态字段模板化。**

---

## 7. 从产品角度，我的总体判断

### 7.1 当前系统已经具备什么

你现在的系统已经不是零散模块，而是一个完整的资产管理骨架：

- 分类
- 位置
- 台账
- 单据
- 盘点
- 维修
- 流水
- 看板
- 附件

这套结构已经足以支撑一套内部固定资产管理系统。

### 7.2 当前系统还缺什么

如果未来要做成更广义的资产平台，还缺：

- 不动产能力
- 财务折旧能力
- 采购来源能力
- 合同能力
- 移动端扫码能力

这些不适合和当前 MVP 一起硬塞进去，应该分阶段做。

### 7.3 最稳的推进顺序

1. 新增资产、单据、盘点、维修页面化
2. 统一操作栏规范
3. 引入资产类型
4. 引入分类字段模板
5. 再考虑不动产和其他扩展类型

---

## 8. 最后结论

如果只说一句话：

**你现在的系统已经够支撑内部固定资产管理，但要继续优化，就应该从“固定表单 + 弹窗操作”升级成“类型化资产 + 模板化字段 + 页面化流程 + 统一操作栏”。**

这条路比继续堆字段更稳，也更适合后续扩展。

## 9. 页面形态决策表

这张表是给研发和产品一起对齐用的，目的很简单：哪些地方保留弹窗，哪些地方必须做成页面，不再靠临时口头约定。

| 场景 | 当前推荐形态 | 最终建议 | 说明 |
|---|---|---|---|
| 分类管理 | 弹窗 + 列表页 | 保持弹窗 | 字段少、树结构明确，弹窗足够 |
| 存放位置 | 弹窗 + 列表页 | 保持弹窗 | 维护成本低，树形编辑适合弹窗 |
| 资产新增 | 轻量弹窗 | 升级为独立页面或向导页 | 字段多，后续还会接模板字段、附件、校验 |
| 资产编辑 | 弹窗或抽屉 | 轻量编辑保留弹窗，复杂编辑转页面 | 快捷编辑和标准编辑分开 |
| 单据新增 | 弹窗 | 升级为独立页面 | 有明细、有流程、有审批上下文 |
| 单据编辑 | 弹窗 | 升级为独立页面 | 与新增同一套页面结构，避免双套维护 |
| 盘点任务新增 | 弹窗 | 升级为独立页面 | 任务范围、明细、差异处理都更适合页面化 |
| 盘点结果处理 | 弹窗 | 保持抽屉或页面 | 如果只做单项确认，可抽屉；批量处理建议页面 |
| 维修单新增 | 弹窗 | 升级为独立页面 | 可能一单多资产，表单复杂度高 |
| 维修审批/确认 | 弹窗 | 保持弹窗 | 动作短，适合快捷处理 |

### 9.1 决策规则

1. 如果一个表单需要同时处理“头信息 + 明细 + 流程 + 附件 + 动态字段”，优先页面化。
2. 如果一个表单只是“少量字段的快捷补录”，可以继续保留弹窗。
3. 如果后续要接分类模板，资产新增必须页面化，否则后面一定返工。
4. 如果一个页面的操作经常超过 3 层，说明该动作应该从弹窗升级为页面或向导页。

### 9.2 页面化的收益

1. 更适合复杂校验。
2. 更适合分步录入。
3. 更适合未来叠加动态字段。
4. 更适合承载附件、明细和审批上下文。

## 10. 动态字段模板草案

这部分是给“分类字段模板化”落地用的。建议不要直接做成完全自由的低代码表单，而是做一套受控模板。

### 10.1 模板目标

1. 让不同分类可以定义自己的专属字段。
2. 让字段支持必填、默认值、校验和展示控制。
3. 让前端能根据模板动态渲染不同组件。
4. 让后续报表和查询仍然保留核心字段，不被模板拖垮。

### 10.2 模板分层

#### A. 核心字段

所有资产都必须有，而且建议继续放在 `asset_info` 中。

例如：

- 资产编码
- 资产名称
- 分类
- 资产状态
- 责任人
- 部门
- 位置
- 原值
- 残值
- 采购日期
- 入库日期

#### B. 分类规则字段

由 `asset_category` 管理，控制资产规则。

例如：

- 是否折旧
- 是否需要序列号
- 是否允许借用
- 是否纳入盘点
- 使用寿命

#### C. 分类扩展字段

只在某些分类下出现的字段，建议通过模板配置。

### 10.3 建议的模板字段

| 字段 | 含义 | 说明 |
|---|---|---|
| field_code | 字段编码 | 全局唯一，供前后端引用 |
| field_name | 字段名称 | 表单展示名称 |
| asset_type | 资产类型 | 固定资产、不动产、低值易耗品等 |
| category_id | 分类 ID | 所属分类，可为空表示类型级模板 |
| field_type | 数据类型 | string / number / date / boolean / object |
| component_type | 组件类型 | input / select / radio / date-picker 等 |
| required_flag | 是否必填 | 控制表单校验 |
| readonly_flag | 是否只读 | 控制编辑状态 |
| searchable_flag | 是否可搜索 | 控制列表筛选项 |
| list_show_flag | 是否展示在列表 | 控制表格列 |
| export_flag | 是否参与导出 | 控制导出字段 |
| dict_type | 字典类型 | select / radio 时使用 |
| default_value | 默认值 | 新增时默认填充 |
| sort_num | 排序号 | 控制展示顺序 |
| group_name | 分组名 | 例如基础信息、权属信息、特有信息 |
| validation_rule | 校验规则 | 正则、长度、范围等 |
| status | 启用状态 | 控制模板是否生效 |
| version_no | 版本号 | 模板升级时便于兼容历史数据 |

### 10.4 推荐的组件映射

| 数据类型 | 推荐组件 |
|---|---|
| string | `ElInput` |
| text | `ElInput` + textarea |
| number | `ElInputNumber` |
| date | `ElDatePicker` |
| datetime | `ElDatePicker` |
| boolean | `ElSwitch` 或 `ElRadioGroup` |
| enum | `ElSelect` / `ElRadioGroup` |
| tree | `ElTreeSelect` |
| user | 人员选择器 |
| dept | 部门选择器 |
| upload | 上传组件 |

### 10.5 模板落地原则

1. 核心字段不要模板化，否则报表会难做。
2. 高筛选频率字段不要只做动态字段，后面要考虑升成正式列。
3. 分类模板变更后，要有版本号，避免历史资产被新模板误伤。
4. 前端动态渲染只负责“渲染和校验”，不要把业务规则全放前端。

## 11. 操作栏统一规范草案

这部分是为前端页面统一交互风格准备的。

### 11.1 统一原则

1. 顶部主操作区使用按钮，保留层级感。
2. 行内操作优先使用文字按钮，减少视觉干扰。
3. 危险操作统一为红色文字或危险按钮。
4. 次级动作收进“更多”菜单。
5. 不同页面的按钮命名保持一致。

### 11.2 顶部操作栏规范

#### 推荐按钮顺序

1. 主按钮
2. 次按钮
3. 业务辅助按钮
4. 危险按钮

#### 推荐例子

- 主按钮：`新增资产`、`新建单据`、`新建任务`
- 次按钮：`导入`、`导出`
- 辅助按钮：`批量处理`、`展开全部`
- 危险按钮：`删除`

#### 交互建议

1. 列表页主操作区尽量固定在顶部，避免长表格滚动后找不到按钮。
2. 当按钮数量过多时，优先把不常用动作收进更多菜单。
3. 不同模块的主按钮尽量不要出现完全不同的表达方式。

### 11.3 行内操作栏规范

#### 推荐按钮顺序

1. 查看
2. 编辑
3. 提交 / 审批 / 完成
4. 更多
5. 删除

#### 风格建议

1. 常规动作统一为文字按钮。
2. 高风险动作使用红色。
3. 审批动作和完成动作可以保留显式按钮，但不要堆太多。
4. 如果一行有超过 4 个动作，建议收进“更多”。

### 11.4 页面级统一约定

1. “新增”统一叫新增，不要有的叫新建、有的叫添加、有的叫创建。
2. “导出”统一叫导出，不要另起别名。
3. “删除”统一叫删除，危险语义保持一致。
4. “查看”统一叫查看，详情页和抽屉页口径一致。

### 11.5 适用边界

这套规范适用于：

- 分类管理
- 存放位置
- 资产台账
- 业务单据
- 盘点任务
- 维修管理
- 资产流水

## 12. 资产新增页面信息架构草案

资产新增页面建议从“一个大弹窗”升级成“独立页面 + 分区布局”。页面的目标不是把字段堆满，而是把录入顺序和业务依赖理顺。

### 12.1 页面目标

1. 支持标准资产一次录入完成。
2. 支持分类切换后动态加载专属字段。
3. 支持附件、责任关系、位置、金额等同步录入。
4. 支持保存草稿和分步补录。

### 12.2 页面布局建议

#### 顶部区域

- 页面标题
- 资产编码状态
- 所属分类状态
- 保存 / 保存并继续 / 取消

#### 主体区域

建议分成 5 个区块：

1. 基础信息
2. 归属信息
3. 价值信息
4. 分类扩展字段
5. 附件与备注

### 12.3 基础信息区

建议字段：

- 资产编码
- 资产名称
- 资产分类
- 资产类型
- 资产状态
- 资产来源
- 品牌
- 型号
- 规格
- 序列号

### 12.4 归属信息区

建议字段：

- 使用部门
- 管理部门
- 责任人
- 存放位置
- 采购日期
- 入库日期
- 启用日期

### 12.5 价值信息区

建议字段：

- 原值
- 残值
- 保修截止日期
- 供应商
- 采购凭证号

### 12.6 分类扩展字段区

这一块由分类模板动态渲染，内容随分类变化。

### 12.7 页面交互规则

1. 先选分类，再加载扩展字段。
2. 分类切换时，如果已填写扩展字段，要提示是否清空重填。
3. 必填校验先做本页校验，再做后端规则校验。
4. 保存草稿时允许部分字段为空，但核心字段不能为空。
5. 保存完成后可以选择继续新增下一条。

### 12.8 适用范围

这个页面适合：

- 标准资产新增
- 批量导入后的补录
- 需要动态字段的资产新增

不适合：

- 快速补登记
- 简单状态修正
- 只改一个责任字段的轻量场景

## 13. 单据新增页面信息架构草案

单据新增页面建议统一成一个通用模板，再按 `order_type` 渲染不同内容。这样以后扩展维修、报废、借用等类型时，不需要再做很多重复页面。

### 13.1 页面目标

1. 统一承载入库、领用、归还、调拨、报废等单据新增。
2. 头信息和明细信息分开处理。
3. 支持按单据类型自动切换字段和明细规则。
4. 支持草稿、提交前预览和保存继续编辑。

### 13.2 页面布局建议

#### 顶部区域

- 单据类型
- 单据编号
- 单据状态
- 保存草稿
- 提交审批
- 取消

#### 主体区域

建议分成 4 个区块：

1. 单据头信息
2. 业务对象信息
3. 资产明细
4. 附件与备注

### 13.3 单据头信息区

建议字段：

- 单据类型
- 单据编号
- 发起部门
- 发起人
- 业务日期
- 单据状态
- 审批流程说明

### 13.4 业务对象信息区

不同单据类型显示不同内容。

#### 入库单

- 来源
- 入库方式
- 关联采购信息

#### 领用单

- 领用部门
- 目标责任人
- 目标位置

#### 归还单

- 归还人
- 归还后状态
- 归还后位置

#### 调拨单

- 来源部门
- 目标部门
- 来源位置
- 目标位置
- 来源责任人
- 目标责任人

#### 报废单

- 报废原因
- 残值
- 处置方式

### 13.5 资产明细区

明细区建议支持：

- 资产选择
- 批量导入明细
- 每条明细的前后状态
- 每条明细的前后责任人
- 每条明细的前后位置
- 每条明细的处理结果

### 13.6 页面交互规则

1. 不同单据类型只显示自己需要的字段。
2. 明细支持批量选择资产。
3. 保存草稿时允许明细未完整，但不能缺少单据类型和发起人。
4. 提交前必须校验明细资产是否重复。
5. 审批后不允许直接改头信息，只能走撤回或作废流程。

### 13.7 适用范围

适合：

- 入库
- 领用
- 归还
- 调拨
- 报废

不适合：

- 简单备注类动作
- 单纯只读详情

## 14. 分类字段模板数据结构草案

这部分是给研发落表和接口使用的，不追求一次性覆盖所有低代码能力，只做“受控模板”。

### 14.1 设计目标

1. 支持分类级字段差异。
2. 支持必填、只读、默认值和校验规则。
3. 支持组件类型自动渲染。
4. 支持模板版本演进。

### 14.2 建议的数据结构

#### 分类字段模板主表

| 字段 | 类型 | 说明 |
|---|---|---|
| template_id | bigint | 主键 |
| asset_type | varchar(20) | 资产类型 |
| category_id | bigint | 分类 ID |
| template_name | varchar(100) | 模板名称 |
| template_version | int | 模板版本 |
| status | char(1) | 启用状态 |
| remark | varchar(255) | 备注 |

#### 分类字段模板明细表

| 字段 | 类型 | 说明 |
|---|---|---|
| field_id | bigint | 主键 |
| template_id | bigint | 所属模板 |
| field_code | varchar(64) | 字段编码 |
| field_name | varchar(100) | 字段名称 |
| field_type | varchar(20) | 数据类型 |
| component_type | varchar(30) | 组件类型 |
| required_flag | char(1) | 是否必填 |
| readonly_flag | char(1) | 是否只读 |
| searchable_flag | char(1) | 是否可搜索 |
| list_show_flag | char(1) | 是否列表展示 |
| export_flag | char(1) | 是否导出 |
| dict_type | varchar(100) | 字典类型 |
| default_value | varchar(255) | 默认值 |
| sort_num | int | 排序 |
| group_name | varchar(100) | 分组 |
| validation_rule | varchar(500) | 校验规则 |
| placeholder | varchar(255) | 占位提示 |
| help_text | varchar(255) | 帮助说明 |
| status | char(1) | 启用状态 |

#### 资产扩展值表或 JSON 字段

两种方式都可以：

1. **JSON 方式**
   - 在 `asset_info` 里增加扩展 JSON 字段
   - 适合一期快速落地

2. **扩展值表方式**
   - 单独存字段值
   - 适合后续做复杂查询和统计

### 14.3 推荐的落地方式

一期建议：

1. 模板定义放数据库表
2. 值存储先用 JSON
3. 高频字段后续再升列

这样可以先把分类扩展能力做出来，又不把查询和报表拖垮。

### 14.4 字段类型建议

| field_type | 说明 | 示例组件 |
|---|---|---|
| string | 普通文本 | 输入框 |
| number | 数值 | 数字输入框 |
| date | 日期 | 日期选择器 |
| datetime | 日期时间 | 日期时间选择器 |
| boolean | 布尔值 | 单选 / 开关 |
| enum | 枚举 | 下拉框 / 单选 |
| tree | 树选择 | 树选择器 |
| user | 人员 | 人员选择器 |
| dept | 部门 | 部门选择器 |
| upload | 附件 | 上传组件 |

### 14.5 模板变更规则

1. 模板版本变化时要保留历史版本。
2. 已有资产的历史值不能因为新模板而丢失。
3. 新模板启用前要先校验字段编码是否重复。
4. 核心字段不要模板化。


不适用的情况：

- 极少量字段的轻量配置页
- 纯只读报表页
- 特殊弹窗确认场景
