## 目标

这份文档把前面的菜单初始化继续往下推进一层，重点解决四件事：

1. 资产模块建议新增哪些角色
2. 每个角色应该绑定哪些菜单和按钮权限
3. `sys_role_menu` 应该怎么写成接近可执行的 SQL 草案
4. 数据范围到底归 `sys_role` / `sys_role_dept` 管，还是归业务 SQL 管

## 基线

当前项目里若依原生表结构已经具备：

1. `sys_role`
2. `sys_role_menu`
3. `sys_role_dept`

其中：

1. `sys_role` 负责角色定义和 `data_scope`
2. `sys_role_menu` 负责角色与菜单权限绑定
3. `sys_role_dept` 只在 `data_scope = 2` 的“自定义数据权限”场景下使用

所以这一轮最重要的原则是：

不要把“菜单授权”和“数据范围”写成一回事。

## 当前建议

### 建议 1：资产模块单独建角色，不要都塞进“普通角色”

若依初始化里已有：

1. `admin`
2. `common`

但资产管理是一个完整业务域，不适合继续把全部权限堆在 `common` 里。更稳的做法是单独新增资产域角色。

### 建议 2：`sys_role_menu` 只表达“能不能做”

例如：

1. 能不能看到“资产台账”页面
2. 能不能点击“单据审批”
3. 能不能执行“处理盘点差异”

这些都属于菜单或按钮权限。

### 建议 3：`data_scope` 只表达“能看谁的数据”

例如：

1. 看全公司
2. 看本部门及以下
3. 看自定义部门

这类事情不应塞进 `sys_role_menu`。

### 建议 4：资产使用人的“本人范围”不要只靠若依通用数据权限

若依原生 `data_scope` 擅长：

1. 全部
2. 自定义部门
3. 本部门
4. 本部门及以下

但它并不天然表达“仅本人资产、仅本人单据”。因此：

1. `asset_user` 可以保留最小菜单权限
2. 真正的“本人范围”必须在资产模块查询 SQL 里补充 `current_user_id` 过滤

## 建议角色清单

建议第一阶段保留下面 5 类角色：

| 角色名 | role_key | data_scope | 说明 |
|---|---|---|---|
| 超级管理员 | `admin` | `1` | 已存在，不新增 |
| 资产管理员 | `asset_admin` | `2` 或 `1` | 中央资产团队，推荐单独角色 |
| 部门主管 | `asset_dept_manager` | `4` | 看本部门及以下，负责审批和差异确认 |
| 资产使用人 | `asset_user` | `2` | 仅作菜单授权，真实范围靠业务 SQL |
| 财务审计 | `asset_auditor` | `1` | 全局查阅、导出、审计 |

说明：

1. `asset_admin` 如果是集团统一资产团队，可直接用 `data_scope = 1`
2. 如果资产管理员只负责部分部门，则用 `data_scope = 2`，并维护 `sys_role_dept`
3. `asset_user` 不建议把“本人范围”硬塞到 `sys_role_dept`

## 推荐授权矩阵

### 1. 超级管理员

不需要单独新增授权脚本。系统本身已经默认拥有全部权限。

### 2. 资产管理员

建议拥有：

1. `资产管理` 一级目录
2. `资产看板`
3. 分类、位置、台账、单据、盘点、流水全部页面
4. `asset:*` 下全部按钮权限

### 3. 部门主管

建议拥有：

1. `资产管理` 一级目录
2. `资产看板`
3. `资产台账` 页面和查询权限
4. `业务单据` 页面、查询、审批、驳回、导出
5. `盘点任务` 页面、查询、差异处理
6. `资产流水` 页面、查询

### 4. 资产使用人

建议拥有：

1. `资产管理` 一级目录
2. `资产看板`
3. `资产台账` 页面、查询
4. `业务单据` 页面、查询、新增、提交

说明：

1. 不建议给普通使用人 `approve / reject / finish / cancel`
2. 也不建议让普通使用人进入分类、位置、盘点、流水页面

### 5. 财务审计

建议拥有：

1. `资产管理` 一级目录
2. `资产台账` 页面、查询、导出
3. `业务单据` 页面、查询、导出
4. `资产流水` 页面、查询、导出
5. 如果业务要求，也可以额外给报废审批权限

select @asset_admin_role_id := role_id from sys_role where role_key = 'asset_admin' limit 1;

select @asset_user_role_id := role_id from sys_role where role_key = 'asset_user' limit 1;

### 2. 回查资产模块菜单根节点

```sql
select @asset_root_id := menu_id
from sys_menu
where parent_id = 0 and path = 'asset'
limit 1;
```

### 3. 给资产管理员绑定菜单

这类角色最适合“全模块授权”。

```sql
insert into sys_role_menu (role_id, menu_id)
select @asset_admin_role_id, m.menu_id
from sys_menu m
where (
  m.menu_id = @asset_root_id
  or m.parent_id = @asset_root_id
  or m.perms like 'asset:%'
)
and not exists (
  select 1 from sys_role_menu rm
  where rm.role_id = @asset_admin_role_id
    and rm.menu_id = m.menu_id
);
```

说明：

1. 这里把一级目录、二级页面、所有按钮一起授权
2. 如果后面新增新的 `asset:*` 按钮，这条思路也便于补齐

### 4. 给部门主管绑定菜单

```sql
insert into sys_role_menu (role_id, menu_id)
select @asset_dept_manager_role_id, m.menu_id
from sys_menu m
where (
  m.menu_id = @asset_root_id
  or m.path in ('dashboard', 'info', 'order', 'inventory', 'event')
  or m.perms in (
    'asset:dashboard:view',
    'asset:info:list',
    'asset:info:query',
    'asset:order:list',
    'asset:order:query',
    'asset:order:approve',
    'asset:order:reject',
    'asset:order:export',
    'asset:inventory:list',
    'asset:inventory:query',
    'asset:inventory:processDiff',
    'asset:event:list',
    'asset:event:query'
  )
)
and not exists (
  select 1 from sys_role_menu rm
  where rm.role_id = @asset_dept_manager_role_id
    and rm.menu_id = m.menu_id
);
```

### 5. 给资产使用人绑定菜单

```sql
insert into sys_role_menu (role_id, menu_id)
select @asset_user_role_id, m.menu_id
from sys_menu m
where (
  m.menu_id = @asset_root_id
  or m.path in ('dashboard', 'info', 'order')
  or m.perms in (
    'asset:dashboard:view',
    'asset:info:list',
    'asset:info:query',
    'asset:order:list',
    'asset:order:query',
    'asset:order:add',
    'asset:order:submit'
  )
)
and not exists (
  select 1 from sys_role_menu rm
  where rm.role_id = @asset_user_role_id
    and rm.menu_id = m.menu_id
);
```

注意：

1. 这个角色即使拿到了 `asset:info:query`，也不代表可以看全量资产
2. 真实返回哪些数据，必须在业务查询里按“本人资产 / 本人单据”继续过滤

### 6. 给财务审计绑定菜单

```sql
insert into sys_role_menu (role_id, menu_id)
select @asset_auditor_role_id, m.menu_id
from sys_menu m
where (
  m.menu_id = @asset_root_id
  or m.path in ('info', 'order', 'event')
  or m.perms in (
    'asset:info:list',
    'asset:info:query',
    'asset:info:export',
    'asset:order:list',
    'asset:order:query',
    'asset:order:export',
    'asset:event:list',
    'asset:event:query',
    'asset:event:export'
  )

and not exists (
  select 1 from sys_role_menu rm
  where rm.role_id = @asset_auditor_role_id
    and rm.menu_id = m.menu_id

```

## `sys_role_dept` 建议

### 资产管理员

如果资产管理员只负责部分组织，才需要落 `sys_role_dept`。

示意写法：

```sql
insert into sys_role_dept (role_id, dept_id)
select @asset_admin_role_id, d.dept_id
from sys_dept d
where d.dept_id in (100, 101, 102)
and not exists (
  select 1 from sys_role_dept rd
  where rd.role_id = @asset_admin_role_id
    and rd.dept_id = d.dept_id
);
```

### 部门主管

一般不需要单独维护 `sys_role_dept`，因为推荐直接使用：

1. `data_scope = 4`

也就是“本部门及以下”。

### 资产使用人

不建议用 `sys_role_dept` 去表达“本人资产”。这是两个维度：

1. `sys_role_dept` 管部门范围
2. “本人资产 / 本人单据”是业务对象归属范围

这部分应交给资产台账和单据查询 SQL 自己处理。

## 推荐落地顺序

真正开始写初始化脚本时，建议顺序如下：

1. 先执行 `sys_menu` 初始化
2. 再执行 `sys_role` 初始化
3. 然后执行 `sys_role_menu` 授权
4. 如果有自定义部门范围，再执行 `sys_role_dept`
5. 最后联调前端按钮显示和后端接口权限

## 当前不建议做的事

### 1. 不建议把所有员工都直接绑定成 `asset_user`

更稳的方式是：

1. 先只给试点部门或试点用户组授权
2. 待“本人范围过滤”联调稳定后，再扩大范围

### 2. 不建议让部门主管拥有 `asset:order:finish`

审批和执行最好分开。否则容易出现“既审批又落账”的边界不清问题。

### 3. 不建议让财务审计默认拥有修改类权限

一期更建议他们保持“查阅 + 导出 + 必要审批”，而不是进入业务执行链路。
