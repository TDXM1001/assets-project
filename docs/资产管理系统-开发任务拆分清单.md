## 目标

这份文档不再继续补设计细节，而是把前面的方案文档正式收敛成可执行的开发任务包，重点解决四件事：

1. 先做什么，后做什么
2. 哪些任务是前置依赖
3. 每一阶段应该交付什么
4. 什么叫“这一阶段真的完成了”

## 当前前提

当前实施前提已经比较明确：

1. 后端基于 `RuoYi-Vue`
2. 前端基于 `art-design-pro`
3. 不使用代码生成器
4. 菜单、角色、表结构、接口分组都已经有设计草案
5. 当前目标是先把固定资产管理 MVP 跑通

所以这份任务清单默认围绕下面这条主线展开：

1. 主数据可用
2. 单据闭环可用
3. 盘点可用
4. 看板与增强可用

## 总体拆分原则

### 原则 1：先铺地基，再做流程

不要一开始就写审批和盘点页面。先把：

1. 数据表
2. 菜单权限
3. 主数据模块

立住，后面流程页才不会返工。

### 原则 2：前后端按“接口成熟度”推进

不是后端全部写完再做前端，也不是前端先空搭页面。而是：

1. 主数据接口 ready，就同步做主数据页面
2. 单据接口 ready，就同步做单据页面

### 原则 3：优先完成业务闭环，不追求页面一次性很满

例如一期单据页的目标是：

1. 能新建
2. 能提交
3. 能审批
4. 能完成

而不是一开始就把所有筛选器、统计卡片、批量操作全补齐。

## 阶段划分

建议拆成 6 个阶段。

## Phase 0：基础准备

### 目标

把实施地基铺平，让后面开发不再反复改方向。

### 前置依赖

无。

### 任务清单

1. 确认 `asset_*` 业务表最终字段版本
2. 确认字典项清单
3. 确认 `sys_menu` 初始化脚本版本
4. 确认 `sys_role` / `sys_role_menu` 初始化脚本版本
5. 确认前后端目录结构和命名规则
6. 清理前端空壳目录 `src/views/asset/ledger` 的后续策略

### 交付物

1. 正式版 DDL SQL 文件
2. 正式版菜单初始化 SQL 文件
3. 正式版角色授权 SQL 文件
4. 文档目录下的实施口径不再冲突

### 完成标准

1. 数据库可以初始化出完整表结构
2. 系统菜单里能看到资产管理目录和子菜单
3. 角色授权脚本可重复执行
4. 团队对命名和目录结构不再有分歧

## Phase 1：主数据可用

### 目标

先把“资产能录、能查、能改”做起来。

### 前置依赖

1. Phase 0 完成

### 后端任务

1. 建 `asset/category` 模块骨架
2. 建 `asset/location` 模块骨架
3. 建 `asset/info` 模块骨架
4. 实现分类树查询和位置树查询
5. 实现资产台账列表、详情、新增、编辑、删除
6. 实现台账导出
7. 预留导入接口入口

### 前端任务

1. 新增 `router/modules/asset.ts`
2. 接入 `category` 页面
3. 接入 `location` 页面
4. 接入 `info` 页面
5. 完成对应 `src/api/asset/*` 文件
6. 完成搜索栏、表格、弹窗统一风格

### 联调任务

1. 校验 `v-auth` 与后端权限标识一致
2. 校验树形数据加载和回显
3. 校验台账页分页、筛选、导出

### 交付物

1. 分类管理页面可用
2. 存放位置页面可用
3. 资产台账页面可用
4. 菜单权限联调通过

### 完成标准

1. 可以在前台新增分类、位置、资产
2. 可以在前台查询和编辑台账
3. 权限不足时按钮会正确隐藏
4. 数据能从页面走完整 CRUD 闭环

## Phase 2：业务单据闭环

### 目标

把最核心的单据流转做通。

### 前置依赖

1. Phase 1 完成

### 后端任务

1. 建 `asset/order` 头表与明细表骨架
2. 实现单据列表、详情、新增、编辑、删除
3. 实现保存草稿
4. 实现提交
5. 实现审批通过
6. 实现审批驳回
7. 实现完成
8. 实现取消
9. 在关键动作里写入 `asset_event_log`
10. 在完成动作里联动更新 `asset_info`

### 前端任务

1. 新增 `order` 页面
2. 完成单据搜索栏
3. 完成新建单据弹窗
4. 完成详情抽屉
5. 完成审批弹窗
6. 完成状态标签与动作按钮区

### 联调任务

1. 验证“新建 -> 提交 -> 审批 -> 完成”闭环
2. 验证“新建 -> 提交 -> 驳回”闭环
3. 验证“完成后资产台账状态同步变化”
4. 验证事件流水落库

### 交付物

1. 单据列表页可用
2. 单据提交流程可用
3. 单据审批流程可用
4. 资产状态联动可用

### 完成标准

1. 入库并直接领用这条流程能完整跑通
2. 跨部门调拨这条流程能完整跑通
3. 驳回、取消等异常分支也能正常工作

## Phase 3：盘点任务可用

### 目标

让“账实差异”开始能被系统处理。

### 前置依赖

1. Phase 2 完成

### 后端任务

1. 建 `asset/inventory` 头表与明细表骨架
2. 实现任务列表、详情、新增、编辑、删除
3. 实现开始盘点
4. 实现批量生成盘点明细
5. 实现结束盘点
6. 实现差异处理
7. 对差异处理结果写入 `asset_event_log`
8. 对需要回写的资产信息同步更新

### 前端任务

1. 新增 `inventory` 页面
2. 完成任务列表和筛选
3. 完成创建任务弹窗
4. 完成详情抽屉
5. 完成差异处理弹窗

### 联调任务

1. 验证盘点任务能生成明细
2. 验证位置不符差异能处理
3. 验证责任人不符差异能处理
4. 验证处理后资产台账和流水同步变化

### 交付物

1. 盘点任务列表页可用
2. 盘点开始与结束动作可用
3. 差异处理可用

### 完成标准

1. “盘点发现位置不符”这条业务流程能完整跑通
2. 账实差异处理后可追溯
3. 主管权限与资产管理员权限边界正确

## Phase 4：只读与聚合能力

### 目标

把管理视角真正补齐，而不只是能做业务动作。

### 前置依赖

1. Phase 3 完成

### 后端任务

1. 完成 `asset/event` 查询与导出
2. 完成 `asset/dashboard` 汇总查询
3. 完成待办、趋势、状态分布接口

### 前端任务

1. 新增 `event` 页面
2. 新增 `dashboard` 页面
3. 完成汇总卡片
4. 完成待办区块
5. 完成状态分布和趋势图

### 联调任务

1. 校验看板统计口径和台账一致
2. 校验导出数据字段正确
3. 校验不同角色看到的看板范围正确

### 交付物

1. 资产流水页可用
2. 资产看板页可用

### 完成标准

1. 管理员能在看板看到真实待办和差异提醒
2. 审计角色能导出流水和台账数据
3. 看板不是空壳页面，而是能反映真实业务状态

## Phase 5：增强与收口

### 目标

把 MVP 从“能用”收口到“比较稳”。

### 前置依赖

1. Phase 4 完成

### 后端任务

1. 完善资产导入
2. 完善附件能力
3. 完善本人范围 / 部门范围过滤
4. 补异常校验和提示文案
5. 补必要的单元测试和集成测试

### 前端任务

1. 完善导入弹窗体验
2. 完善空状态、错误态、加载态
3. 完善详情抽屉信息密度
4. 收口重复组件和重复状态映射

### 联调任务

1. 回归三条核心业务流程
2. 回归菜单权限与按钮权限
3. 回归不同角色的数据范围

### 交付物

1. 更稳的导入导出体验
2. 更稳的权限与数据范围表现
3. 一版可交付的 MVP

### 完成标准

1. 主流程不再依赖人工改数据库兜底
2. 页面异常态有明确反馈
3. 角色切换后不会看到明显越权数据

## 横向任务包

除了按阶段推进，还建议维护 4 条横向任务线。

### 1. SQL 与初始化脚本线

包括：

1. DDL
2. 字典 SQL
3. 菜单 SQL
4. 角色授权 SQL

### 2. 后端骨架线

包括：

1. Controller
2. Service
3. Mapper
4. Mapper XML
5. BO / VO / Entity

### 3. 前端骨架线

包括：

1. Router
2. API
3. 页面
4. 弹窗 / 抽屉模块
5. 权限接线

### 4. 联调与验收线

包括：

1. 权限验证
2. 三条核心流程验证
3. 数据范围验证
4. 导入导出验证

## 推荐任务颗粒度

为了避免任务太大难追踪，建议控制在下面这个颗粒度：

### 合适的任务粒度

1. “完成分类管理后端骨架”
2. “完成资产台账前端列表页”
3. “完成单据提交流程后端逻辑”
4. “完成盘点差异处理弹窗”

### 不合适的任务粒度

1. “做完整个资产管理系统”
2. “做完后端”
3. “做完前端”

## 推荐排期视角

如果按一个人主导、全手写方式推进，建议按下面节奏理解：

### 第一周

1. Phase 0
2. Phase 1

### 第二周

1. Phase 2

### 第三周

1. Phase 3

### 第四周

1. Phase 4
2. Phase 5 的收口工作

这不是死排期，而是告诉你：

1. 真正花时间的不是主数据 CRUD
2. 真正的风险集中在单据、盘点、权限和数据范围

## 当前不建议做的事

### 1. 不建议 Phase 1 就去做复杂看板

这会让你看起来很忙，但对资产系统真正跑通帮助不大。

### 2. 不建议在主流程没跑通前做很多细节抛光

例如：

1. 高级筛选
2. 自定义列
3. 花哨图表
4. 很复杂的批量操作

### 3. 不建议把测试留到最后

至少每完成一个阶段，就要把该阶段的闭环回归一遍，否则问题会堆到后面爆。

## 最终验收口径

如果你要判断“这套 MVP 到底算不算完成”，建议用下面 5 条：

1. 资产可以建账、查询、编辑、导出
2. 领用/调拨类单据可以走完草稿、提交、审批、完成
3. 盘点任务可以创建、执行、发现并处理差异
4. 事件流水可以追溯关键动作
5. 不同角色进入系统后，看到的页面、按钮和数据范围基本正确

## 后续建议

如果这份拆分清单方向没问题，下一步就不一定还要继续写设计文档了，更自然的是直接进入两种动作之一：

1. 按 Phase 0 开始产出正式 SQL 文件
2. 按 Phase 1 开始手写后端和前端骨架
